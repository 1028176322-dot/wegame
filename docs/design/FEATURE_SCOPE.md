# 功能范围说明书 (Feature Scope) — 塔防微信小游戏

> 状态：`v0.1-draft`，待 Decision Owner 确认范围边界（未提交）

---

## 1. 范围方法论（为什么这么拆）

- **核心循环优先**：先保证 MVP 的"建塔→养塔→撑波→清屏"可独立好玩，再叠加外层。外层堆得再花，核心循环不好玩也救不回来。
- **每个功能回答两个问题**：玩家*感受到什么*？玩家*在做什么决策*？答不上来的功能不进范围。
- **分层标记**：
  - `MVP (v1.0)`：核心循环闭环必需，玩家第一局就能体验
  - `增强 (v1.x)`：留存/深度，首版可无但不久必补
  - `探索 (stretch)`：依赖平台合规或高不确定，先做可行性判断
  - `不做 (out)`：明确排除，记录理由避免范围漂移
- **优先级对齐治理**：MVP = P0/P1，增强 = P2，探索 = P3。

## 2. 核心玩法功能（MVP v1.0，P0/P1）

| ID | 功能 | 玩家体验 / 决策 | GDD 对应 | 优先级 |
|----|------|----------------|----------|--------|
| F1 | 环形地图与多圈寻路 | 看到怪绕圈多趟，越往后越紧张；决策：塔放哪圈 | §4 地图 | P0 |
| F2 | 塔位 + 一键建造 | 单指点格即建；决策：先建哪种、建哪 | §5.1 Build | P0 |
| F3 | 首发塔种（箭/炮/冰/风塔〔新风种，DO 已确认〕，解锁制） | 风塔为第二类控制（击退），与冰塔 combo；其余塔种经元进度解锁，决策：木喂谁、何时起风塔 | §5.7 塔种 / §5.8 技能 | P0 |
| F4 | 养塔升级（喂木，指数成长） | 看塔从弱到一发清屏的爽点本体；决策：木喂谁 | §5.2 养 | P0 |
| F5 | 双货币经济（金↔木 + 应急兑换〔session 木=怪掉+应急〕） | 金建塔、木养塔；木主源=怪掉，应急兑换为受限兜底；决策：养哪座、何时应急 | §6 经济 / §5.8 | P0 |
| F6 | 波次与敌人（护甲/克制 + Boss） | 每波压力递增，Boss 前做取舍；决策：这波怎么配塔 | §5.6 波次 | P0 |
| F7 | 战斗结算（目标/弹道/伤害/状态） | 命中飘字、击杀反馈；决策：靠数值还是靠布局 | §5.7 + §5.8 | P0 |
| F8 | 漏怪与 Lives | 漏怪扣命，0 命失败；制造"别漏"的紧张 | §5.5 漏怪 | P0 |
| F9 | HUD / 操控（2x、暂停、单指） | 一指可玩；决策：加速还是稳一手 | §9 适配 | P1 |
| F10 | 新手引导（P3 零门槛） | 首波保送、机制逐个低压引入；决策：在引导里学取舍 | §7 引导 | P1 |
| F11 | 本地存档（wx.setStorage） | 进度不丢；决策：无（系统层） | 工程 | P1 |
| F12 | 胜负结算 / 重开 | 一局有头有尾，留"再来一局"钩子 | §5.7 胜负 | P1 |

> **MVP 验收标准（设计侧）**：无引导辅助下，新玩家首局能建塔、养塔（靠怪掉木）、撑过前 3 波、理解风塔/技能作用，且核心循环"好玩"成立。

> **等级系统（F45）验收（设计侧）**：每局结算正确产出 XP 并可能升级；升级后新建塔自动享受当前等级加成（**不累加，单行**）；达 `unlock_config.required_level`（S29）正确解锁对应塔种/功能并弹通知；`player_level`/`current_xp` 持久化跨局保留；等级增益普惠全塔种（无主导策略）、全自动零操作（无认知过载）。

## 6. 明确不做（Out of Scope）

| 功能 | 理由 |
|------|------|
| 多人实时合作 | 小游戏以异步为主，实时联机成本/合规高，且破坏短局节奏 |
| 好友助战（实时/借塔） | DO 已裁定不做：异步为主，成本/合规高，破坏短局节奏（见 §7） |
| 超长硬核局 | 微信小游戏场景为碎片化短局（3–5min），长局不符平台习惯 |
| PC / 主机 / 跨平台移植 | 当前范围锁定微信小游戏，跨平台不在本期 |
| 自定义关卡编辑器 / UGC | 高不确定、高合规风险，列为探索项而非承诺 |
