## A 域 · 核心战斗域（Fun Hypothesis 载体）

> 这一域是"看自己养的塔从弱到一发清屏"爽点的全部承载体。任何系统如果不能为这个爽点服务或支撑它，都应被质疑。

### S1 地图系统 (Map System)
**关联 F 码**：F1 · **GDD**：§4 地图 · **层级**：MVP / P0

- **目的**：提供环形多圈路径与塔位，让"怪物绕圈多趟、养成塔越养越值"的绿圈精髓成立。
- **玩家体验目标**：一进场就看出"怪会绕好几圈"，自然思考"塔放内侧还是外侧、放哪圈"。
- **子系统 / 功能模块**：
  - 环形路径数据（路点序列、圈数、每圈长度）
  - 塔位网格（可建格 vs 障碍格，塔位占用状态）
  - 寻路 / 移动插值（怪物沿路点匀速移动，多圈循环）
  - 地图渲染层（路径、塔位、可建高亮、范围圈）
- **关键机制**：环形路径使同一怪物被多座塔多次命中；内圈塔"见怪次数"多，养成塔越养越值。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 圈数、每圈长度、塔位总数、可建比例、路径移动速度。
- **依赖系统**：S2（塔位占用）、S5（命中判定）、S19（地图资源加载）。
- **边界 / 明确不做**：不做多地图同屏、不做可破坏地形（v1.0 固定单环形图，多关卡见 S14）。
- **验收信号**：怪物正确绕满 N 圈到达终点；塔位高亮与实际可建一致；内圈塔 DPS 利用率显著高于外圈（纸面模拟验证）。

### S2 建筑（塔）系统 (Building / Tower System)
**关联 F 码**：F2 F3 F4 · **GDD**：§5.1 Build / §5.2 养 / §5.7 塔种 · **层级**：MVP / P0

- **目的**：这是玩家最核心的操作系统——选格、建塔、养塔、卖塔，所有策略决策在此发生。
- **玩家体验目标**：单指点格即有塔立起；喂木后看它肉眼可见地变强；选中塔弹出技能按钮、养塔点亮被动是招牌惊喜。
- **子系统 / 功能模块**：
  - **建造子模块（F2）**：塔位选中 → 塔种选择面板 → 扣金建塔 → 占用塔位；一键取消/重试。
- **塔种库（F3）**：首发 4 种（箭 / 炮 / 冰 / **风塔〔新风种，DO 已确认〕**），解锁制；其余 3 种（魔 / 毒 / 电）走 S11 元进度解锁。
- **养塔子模块（F4）**：选中已建塔 → 喂 **session 木**升级 → 等级↑、属性指数成长（DPS/范围/特效）；木来自怪掉(S04)+应急兑换(S03)，不持久化、每局重来。
  - **售卖 / 退款**：卖塔返还部分金（防卡死），占用塔位释放。
  - **目标 / 索敌模式**：单体 / 最前 / 最强 / 范围，影响策略深度（MVP 可仅"最前"，增强补多模式）。
- **关键机制**：风塔占塔位作**第二类控制（击退/位移）**，与冰塔（减速）形成 combo，制造"风把怪吹回让炮塔多打"的取舍；养塔指数成长是爽点本体；技能(S28)按养塔等级解锁被动，强化养成。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 各塔建造成本、每级喂木量、升级成长系数（建议 1.5–2.0x）、售卖返还率（建议 50–70%）、被动解锁等级（S28）、风塔击退参数（[PLACEHOLDER]）。
- **依赖系统**：S3（金/木扣减；木=session drop）、S1（塔位占用）、S5（塔开火）、S28（技能入口/等级）、S9（引导期低压引入）。
- **边界 / 明确不做**：不做塔融合/合成（超出 v1.0 复杂度）；索敌多模式为增强项。
- **验收信号**：建/养/卖三步在 2 点击内可达；养塔后单发伤害肉眼可感提升；无金时可建状态正确置灰。

### S3 经济系统 (Economy System)
**关联 F 码**：F5 · **GDD**：§6 经济 · **层级**：MVP / P0

- **目的**：用金（建塔主力）+ 木（养塔主力）双币，制造"要不要应急兑换、兑多少木"的持续决策张力，并防通胀。
- **玩家体验目标**：每一波结束都在算"这波金够建还是该应急兑换养塔"；金木兑换率是脑子里绷着的第一根弦。
- **子系统 / 功能模块**：
  - **货币容器**：金、木余额与变动事件（来源/ sinks 统一经此）。
  - **兑换子模块**：金→木固定/浮动兑换率（第一调优杆）；木→金（通常禁止或高损，防刷）。
  - **来源（Sources）**：击杀赏金（金）、**怪掉木（session 木，主源 S04）**、波次结算（金）、Boss 奖励（金）、**应急兑换（金→木，受限 S03）**。
  - **消耗（Sinks）**：建塔（金）、养塔（木）、售卖返还（负 sink）。
  - **通胀检测**：按 GDD §6，监控"每活跃玩家每日 session 木产出"，超阈值触发平衡 pass。
- **关键机制**：双币(session)强制玩家在"扩张（金）"与"养成（木）"间分配；**木主源=怪掉**，应急兑换（金→木）为受限特例（次数/触发条件/较差汇率），不能成为主源。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 初始金/木、兑换率、击杀赏金曲线、波次结算公式、通胀阈值（币/人/日）。
- **依赖系统**：S2（建/养扣币）、S4（波次奖励）、S24（防作弊护经济）、S21（远程调率）。
- **边界 / 明确不做**：F19 广告/分享额外产出在开关 off 时**不接入此系统**（零破环产出）；不做玩家间交易。
- **验收信号**：纸面模拟 100 局无币无限增长（无死循环 source）、无早期币荒卡死（可解性）；兑换率调整即时生效。

### S4 波次系统 (Wave System)
**关联 F 码**：F6 · **GDD**：§5.6 波次 · **层级**：MVP / P0

- **目的**：编排敌人出场节奏与类型，制造逐波递增的压力与"Boss 前做取舍"的结构性决策点。
- **玩家体验目标**：每波来袭前看得到"这波是什么怪、有没有护甲"，据此决定这波怎么配塔。
- **子系统 / 功能模块**：
  - **波次配置表**：每波怪物种类、数量、出场间隔、路径圈数。
  - **护甲 / 克制**：怪物护甲类型（物/魔/毒等）对应塔种克制关系（箭克轻甲、炮克重甲、冰减速、毒持续）。
  - **Boss 波**：高血量 + 特殊机制（如减疗/加速），前置于难度跃迁点。
  - **波间节奏**：准备期（可建/养）+ 出怪期，准备期长度即决策窗口。
- **关键机制**：克制关系使"塔种搭配"成为真决策而非数值堆砌；Boss 波是单局情绪高点。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 总波数、每波强度曲线、护甲减免%、Boss 血量/机制、准备期时长。
- **依赖系统**：S5（敌人属性与受击）、S2（塔种克制）、S8（结算按波数）。
- **边界 / 明确不做**：不做随机无限波（v1.0 固定波表，无尽模式属探索）；多关卡波表见 S14。
- **验收信号**：玩家能在出怪前预判威胁并调整；存在"纯堆数值也过不去"的克制关卡（证明决策有效）。

### S5 战斗系统 (Combat System)
**关联 F 码**：F7 · **GDD**：§5.7 + §5.8 战斗/状态 · **层级**：MVP / P0

- **目的**：把塔的属性、怪物属性、弹道与状态效果，结算成"命中→伤害→反馈"的闭环。
- **玩家体验目标**：每次命中都有飘字与击杀反馈，数值差距通过打击感被"看见"。
- **子系统 / 功能模块**：
  - **弹道 / 命中**：发射、飞行、命中判定（单体/溅射/范围）。
  - **伤害结算**：基础伤害 × 克制系数 × 养塔加成 − 护甲减免。
  - **状态效果**：冰（减速）、毒（持续 DOT）、电（连锁）、炮（溅射）。
  - **反馈层**：飘字、击杀粒子、命中音（表现见 S23/S36 打击感）。
- **关键机制**：克制系数 + 养塔加成让"同一座塔"在不同时期手感天差地别（爽点放大器）。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 各塔基础 DPS、弹速、状态时长/强度、溅射半径、连锁跳数。
- **依赖系统**：S2（塔属性）、S4（敌人属性）、S6（漏怪触发）、S23（音效）。
- **边界 / 明确不做**：不做物理真实弹道（小游戏用简化直线/抛物线）；不做起跑线随机。
- **验收信号**：高养塔塔一发清屏的手感成立；克制关系在数值上可验证（纸面伤害表）。

### S30 属性系统 (Attribute / Stat System)
**关联 F 码**：（战斗基础·跨系统） · **GDD**：§5.7 战斗/克制 · **层级**：MVP / P0

- **目的**：全游戏属性（塔/敌）的**唯一真理源（single source of truth）**——S31 敌人 / S32 关卡 / S33 状态都必须引用它，杜绝属性散落。
- **玩家体验目标**：玩家在图鉴(S16)/塔信息卡看到的每个属性、每次伤害飘字颜色，都来自同一份定义。
- **子系统 / 功能模块**：
  - **属性清单**：塔（dmg/range/atk_speed/projectile_speed/armor_type[恒=none]）+ 敌（hp/armor_type/move_speed/special_behavior）。
  - **枚举**：伤害类型 `physical/magic/poison/control`；护甲类型 `none/light/heavy/magic_immune/air`。
  - **伤害×护甲矩阵（克制）**：对齐 GDD §5.6（物理克轻甲、炮溅射克重甲、魔法无视护甲但魔免×0、毒克高HP、控制全克无伤）。
  - **铁律合成公式**：`tower_effective = base × growth(S02) × player_level_mult(S29 单行不累加) × skill_mod(S28) × buff_mod(S33)`。
- **关键数据 / 调优杆**：克制系数（×1.5/×0.5/×0/×1）、敌属性关卡/波次缩放因子、属性极值钳制。
- **依赖系统**：被 S02/S04/S05/S28/S29/S31/S33 引用（无上游依赖，纯定义层）。
- **边界 / 明确不做**：不定义具体塔/敌数值，只定义 schema 与克制；不做数值平衡（落 balance/S30）。
- **验收信号**：任一属性/克制改动单点生效，全游戏无散落同名定义；图鉴与实战数值一致。

### S31 敌人系统 (Enemy / Monster System)
**关联 F 码**：F6（敌人半边） · **GDD**：§5.5/§5.6 · **层级**：MVP / P0

- **目的**：敌人实体的**唯一定义源**——`enemy_config`（按 enemy_id 唯一）+ `enemy_drop`（逐敌掉木，session 木主源），S04 波表与 S16 图鉴均消费本表。
- **玩家体验目标**：每只怪有清晰原型与弱点，图鉴一眼看懂"谁克它"。
- **子系统 / 功能模块**：
  - **敌人原型**：普通(轻甲)/重甲/空军(air)/魔免(magic_immune)/Boss + poison(高HP肉盾，弱毒)；每类给出 S30 属性初值与被克塔种。
  - **特殊行为**：空军(上层 z)、魔免(魔法×0)、Boss(高HP+加速/减疗)。
  - **掉木率表**：每敌 wood 掉落概率/量（session，不持久化），Boss 额外掉木。
- **关键数据 / 调优杆**：HP/速度/掉木率/量、Boss 机制、敌属性关卡/波次缩放（引用 S30/S32）。
- **依赖系统**：S30（属性 schema）、S04（波表引用 enemy_id）、S03（木 drop）、S32（出怪组合）。
- **边界 / 明确不做**：不定义波次节奏（S04）、不定义地图（S1）；只定义"敌人是什么"。
- **验收信号**：enemy_id 唯一；S04 波表/S16 图鉴只读消费本表，无重复定义。

### S32 关卡配置系统 (Stage / Level Config System)
**关联 F 码**：F17（内容层） · **GDD**：§5.5/§5.6 · **层级**：MVP / P1

- **目的**：关卡**内容数据**的唯一定义源（`stage_config`）——S14 选关壳的数据层，避免"选关 UI 有了、关卡内容却没定义"。
- **玩家体验目标**：每关是不同怪组合/路径/Boss 的新谜题，不是同一张图刷数值。
- **子系统 / 功能模块**：
  - **stage_config**：stage_id / map_variant(引 S1) / wave_table(引 S4) / enemy_composition(引 S31 enemy_id 逐波) / boss_schedule / difficulty_mod(hp_scale,speed_scale) / reward / unlock_level(接 S29)。

### S29 玩家等级系统 (Player Level System)
**关联 F 码**：F45 · **GDD**：§5.9 玩家等级系统 · **层级**：增强 / P2

- **验收信号**：每局结算正确产 XP 并可能升级；升级后新塔自动享当前等级加成（不累加）；达等级门槛正确解锁对应功能并弹通知。

## C 域 · 平台工程运营域（地基与保障）

> 这一域玩家看不见，但缺一个游戏就上不了架或活不久。微信小游戏特有坑（分包/生命周期/合规）都在这里。

### S18 存档系统 (Save System)
**关联 F 码**：F11 F32 · **层级**：MVP / P1

- **目的**：用 `wx.setStorage` 持久化进度，跨会话不丢；结构化 schema + 版本迁移防坏档。
- **玩家体验目标**：关了再开，进度、解锁、签到全在；更新版本不丢档。
- **子系统 / 功能模块**：
  - **Schema 定义**：玩家档 = 元进度 + 设置 + 签到 + 最佳成绩（data-config 当前 unassessed，待定结构）。
  - **读写接口**：统一 save/load，防并发写。
  - **版本迁移**：schema 版本号 + 迁移脚本（旧档→新档）。
  - **容错**：坏档检测 → 回退/重置提示，不崩。
- **关键机制**：单一存档入口，所有系统经此读写，避免散落。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 存储上限（微信约 10MB 本地）、迁移策略。
- **依赖系统**：全系统（统一落档）；S20（生命周期触发存）。
- **边界 / 明确不做**：F42 云存档暂不做（合规后议）；不做多档位。
- **验收信号**：强杀重开进度完整；模拟旧版档升新版不丢数据。

**关联 F 码**：F30 F34 F35 · **层级**：MVP / P1

- **目的**：微信主包体积限制下，把美术/音频分包远程加载，MVP 即规划，否则后期返工。
- **玩家体验目标**：启动快、不卡在 loading；首次进关才下该关资源。
- **子系统 / 功能模块**：
  - **分包规划**：主包（代码+首关）/ 分包（其余关、音频、图集）。
  - **资源管线（F34）**：塔/怪/地图美术格式（图集/骨骼）、微信压缩规范。
  - **音频资产（F35）**：BGM/音效格式与分包。
  - **加载管理**：预载关键资源、按需分包、loading 反馈。
- **关键机制**：分包 = 主包体积达标 + 体验不卡的前提。
- **关键数据 / 调优杆**：`[PLACEHOLDER]` 主包上限（微信约 4MB 主包）、分包大小档、预载策略。
- **依赖系统**：S1（地图资源）、S23（音频）、构建链（F23）。
- **边界 / 明确不做**：不做运行时下载代码包（受平台限制）；不做 P2P 资源。
- **验收信号**：首屏 loading ≤可接受时长；主包体积达标过审。
