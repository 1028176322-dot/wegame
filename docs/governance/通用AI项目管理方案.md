# 通用 AI 项目管理方案

> **版本**: 1.0
> **文档状态**: current / reusable-methodology
> **适用范围**: 软件、游戏、数据、内容、研究、设计、自动化及混合型项目
> **适用团队**: 人工团队、单AI辅助、多AI并行、人机混合团队
> **使用方式**: 将本文件复制到目标项目，替换第1节变量，并按第20节建立最小治理文件集

## 0. 方案目标

本方案解决以下常见问题：

- 文档很多，但不知道哪份代表当前决定。
- 任务执行时间过长，对话上下文压缩后发生范围漂移。
- 多个人或多个AI同时修改文件，互相覆盖、重复实现或破坏未提交工作。
- 把计划、当前事实、目标状态和完成状态混为一谈。
- 测试、审查和验收缺少可复核证据。
- 项目记忆、Skill、脚本和自动化工具持续累积但无人治理。
- 删除、迁移、配置、密钥和发布操作缺少安全边界。
- AI为了“完成任务”而扩大范围、修改测试、弱化门禁或伪造PASS。

核心思想：

> 对话只用于协作，磁盘文件才是任务状态；计划只描述意图，证据才证明完成；并行只发生在不重叠的文件范围，集成始终串行。

### 0.1 适用边界

本方案管理“如何决定、如何执行、如何证明、如何恢复”，不替代具体项目的产品设计、技术架构、法律意见或行业标准。项目必须在通用规则之上增加领域附录，例如医疗、金融、未成年人、支付、隐私、硬件安全或应用商店规则。

以下情况应裁剪而不是机械照搬：

- 无Git的纯内容项目可用对象存储版本号替代commit SHA，但仍需不可变版本锁。
- 单人短期实验可以合并部分角色，但C3风险仍需外部复核。
- 只读分析任务不必创建完整写入租约，但必须声明不会修改外部状态。
- 临时原型可以降低文档数量，不能降低秘密、数据、删除和发布安全门。

### 0.2 成功标准

采用本方案后，任意新成员或新AI应能在不读取旧聊天记录的情况下回答：

1. 当前有效的产品、架构、进度和发布决定在哪里。
2. 当前有哪些任务正在写哪些文件。
3. 某个任务允许改什么、禁止改什么、执行到了哪里。
4. 哪些结果真实通过，哪些没有运行或仍被阻断。
5. 上下文丢失后应从哪个checkpoint继续。
6. 谁能审查、集成、验收、发布和回滚。

### 0.3 不在本方案中自动发生的动作

阅读或采用本方案不自动授权：创建外部资源、发送消息、购买服务、安装依赖、修改生产数据、删除文件、提交代码、推送远程、发布版本或重写历史。这些动作必须由具体任务单独授权。

## 1. 项目初始化变量

将下列变量写入项目入口规则，避免不同执行者自行猜测：

| 变量 | 含义 | 示例值 |
|---|---|---|
| `<PROJECT_NAME>` | 项目名称 | Example Project |
| `<REPO_ROOT>` | 仓库根；文档中只使用相对路径 | `.` |
| `<PRIMARY_PRODUCT_ROOT>` | 主产品目录；没有则为`.` | `app/` |
| `<DEFAULT_BRANCH>` | 默认分支 | `main` |
| `<PRIMARY_PLATFORM>` | 当前默认交付平台 | Web / Desktop / Mobile / General |
| `<PACKAGE_MANAGER>` | 依赖工具 | npm / pnpm / pip / cargo / none |
| `<TEST_COMMAND>` | 项目测试入口 | 项目自定义 |
| `<VALIDATE_COMMAND>` | 全量门禁入口 | 项目自定义 |
| `<RELEASE_OWNER>` | 发布批准者 | 人工负责人 |
| `<INTEGRATOR>` | 当前唯一集成者 | 人工或指定AI |

变量未知时应写`unassessed`或`NOT_IMPLEMENTED`，不得凭经验填充。

### 1.1 初始化步骤

1. 只读识别仓库根、产品根、版本控制、构建工具和运行环境。
2. 读取已有入口、README、贡献指南、CI、包管理文件和安全规则。
3. 运行可用的只读基线命令，记录工具缺失和失败，不立即修复。
4. 与Decision Owner确认项目目标、默认平台、交付物和发布责任。
5. 把已确认变量写入`PROJECT_RULES.md`或机器配置。
6. 为未确认变量建立决策ID、Owner和截止条件。
7. 生成初始化报告，绑定输入文件SHA和HEAD。

### 1.2 项目画像

初始化时至少选择：

| 维度 | 可选值 |
|---|---|
| 生命周期 | prototype / active-development / maintenance / migration / sunset |
| 数据敏感度 | public / internal / confidential / restricted |
| 发布模式 | none / manual / CI-CD / regulated |
| 运行环境 | local / cloud / device / embedded / mixed |
| 团队模式 | single-human / human-AI / multi-AI / multi-team |
| 可逆性 | fully-reversible / partially-reversible / irreversible |

画像决定默认风险等级。例如restricted数据、生产迁移、不可逆操作和受监管发布默认C3。

### 1.3 初始化完成门

- 必填变量没有空字符串或示例占位符。
- 仓库根和产品根没有歧义。
- 测试、验证、发布命令均为真实命令或明确`NOT_IMPLEMENTED`。
- 当前Decision Owner、Integrator、Release Owner可识别。
- 项目画像和敏感度已登记。
- 项目入口文件能路由到权威索引和任务系统。

## 2. 治理原则

### 2.1 十条基本原则

1. **单一事实源**：同一领域最多一份current入口。
2. **最小授权**：只执行用户明确请求和任务卡覆盖的工作。
3. **范围冻结**：写入前冻结目标文件、动作、SHA和验收条件。
4. **事实分离**：当前事实、目标方案、迁移状态、历史材料分别管理。
5. **证据优先**：没有命令结果、文件SHA或人工验收，就不能声称完成。
6. **失败真实**：`NOT_RUN`和`NOT_IMPLEMENTED`不能写成PASS。
7. **并行隔离**：同一文件同一时间只能有一个写入者。
8. **集成串行**：共享入口、总索引、构建配置和发布只能由唯一集成者处理。
9. **破坏性操作默认禁止**：删除、移动、重命名、历史重写先dry-run并单独授权。
10. **规则可执行**：重要规则必须逐步转为自动检查和负向测试。

### 2.2 不接受的完成方式

- 修改、删除、跳过失败测试以取得绿色结果。
- 关闭lint、typecheck、验证脚本或扩大忽略列表。
- 把预存失败当作没有风险。
- 通过批量格式化隐藏实际变更。
- 用新文档覆盖旧冲突，但不声明替代关系。
- 用聊天摘要代替冻结任务卡。
- 由执行者同时扮演自己的独立Reviewer和最终验收者。

### 2.3 规则生命周期

每条强制规则应具有稳定ID、级别、Owner、适用范围、检查方法、例外流程、引入日期和复查触发器。规则状态使用：`proposed / active / deprecated / retired`。

规则变更必须回答：

- 要防止的真实失败是什么。
- 正向行为和负向行为分别是什么。
- 能否自动检测；不能自动检测时谁人工检查。
- 对现有仓库有多少历史命中。
- 是立即阻断还是先建立只减不增基线。
- 如何回滚错误规则。

规则不得在一次业务修复中顺手修改。门禁规则和业务修复同时变化会使验证失去可信基准，应拆成独立任务或由独立Reviewer明确批准。

### 2.4 例外与风险接受

任何例外必须是有限、可追踪、可撤销的，不允许口头永久豁免。例外记录至少包含：规则ID、精确对象、原因、风险、批准者、到期日、补偿控制、关闭任务和验证证据。

例外过期、对象SHA变化或范围扩大时自动失效。新的违规不得复用旧对象的例外ID。

### 2.5 治理停止条件

- 要求执行者伪造PASS、删除证据或隐藏风险。
- 规则来源冲突且没有可识别裁决者。
- 高风险动作没有Owner、回滚或授权。
- 规则修改会让当前验证结果失效，却仍要求继续验收。
- 自动化工具输出与磁盘事实不一致且无法解释。

## 3. 权威体系与单一事实源

### 3.1 权威优先级

建议采用以下顺序：

1. 用户或业务负责人在当前任务中的明确决定。
2. 仓库根项目规则和安全规则。
3. 权威索引指定的领域唯一current文档。
4. 已接受的架构决策记录、治理方案和发布政策。
5. 当前任务的冻结Change List。
6. supporting材料。
7. historical、superseded、deprecated材料。

两个同级current发生冲突时，必须停止该决策分支并申请裁决；禁止按文件日期、文件名、修改时间或“看起来更新”自动选赢家。

### 3.2 领域权威索引

建立`docs/governance/AUTHORITY_INDEX.md`，至少覆盖：

- 产品与需求
- 技术架构
- 数据与配置
- 用户体验/内容/资产
- 测试与质量
- 平台与发布
- 安全与合规
- 项目进度
- AI协作与治理

每个领域至少记录：

| 字段 | 说明 |
|---|---|
| Domain | 稳定领域ID |
| Current | 唯一current入口；可为`unassessed` |
| Supporting | 解释、证据和专题材料 |
| Superseded | 被替代的历史入口 |
| Owner | 维护者 |
| Baseline Date | 当前事实基线日期 |
| Review Trigger | 什么变化会触发重审 |

### 3.3 current文档元数据

每份current文档建议包含：

```text
doc_status: current
view_type: current_state / target_state / mixed / registry / policy
scope:
owner:
baseline_date:
evidence_level:
supersedes:
superseded_by:
review_trigger:
```

`mixed`文档必须在正文中明确区分“当前”“目标”“迁移中”。

### 3.4 权威冲突处理协议

发现冲突时建立`DECISION_CONFLICT_<ID>.md`，记录：

```text
Conflict ID
Domain
Source A + SHA + exact claim
Source B + SHA + exact claim
Operational impact
Safe actions that may continue
Actions frozen
Decision owner
Options and trade-offs
Final decision
Documents to synchronize
```

冲突期间只冻结依赖该决定的动作；无关的只读分析和安全检查可以继续。裁决完成后必须同步current、权威索引、进度、任务卡和自动门禁，不能只在聊天中回复结论。

### 3.5 决策记录与ADR

满足任一条件时建立ADR：公共架构方向、跨模块依赖、数据模型、兼容策略、平台选择、安全边界、不可逆迁移、成本显著的工具/供应商选择。

ADR至少包含背景、决定、候选方案、取舍、影响、迁移、回滚可能性、复查触发器和状态。ADR记录“为什么”，current设计记录“现在是什么”，两者不能互相替代。

### 3.6 current变更流程

1. 创建Change List并冻结旧current和权威索引SHA。
2. 列出所有消费者：任务、进度、门禁、记忆、Skill、发布文档。
3. 修改current或指定新的current。
4. 给旧入口添加superseded关系，不静默删除。
5. 同步权威索引和消费者。
6. 运行唯一性、链接、元数据和口径检查。
7. 由领域Owner验收。

一个current变更未同步权威索引时，不得进入accepted。

## 4. 通用目录结构

建议的最小结构：

```text
PROJECT_RULES.md                    # 所有执行者的唯一启动入口
docs/
  governance/
    AUTHORITY_INDEX.md
    PROJECT_MANAGEMENT.md
    DOCUMENT_RULES.md
    PROGRESS_RULES.md
    ENGINEERING_RULES.md
    SECURITY_RULES.md
  decisions/                       # ADR/决策记录
  progress/
    _index.md
  tasks/
    coordination/
      INDEX.md
      active/
      baselines/
      checkpoints/
      handoffs/
      findings/
    reports/
  memory/
    MEMORY_SCHEMA.md
    inbox/
    topics/
  skills/
    skill_registry.json
    install_lock.json
    trigger_matrix.md
tools/
  validators/
  maintenance/
  generators/
archive/
deprecated/
experimental/
```

目录名称可以适配项目，但职责不能混淆：生成器不承担验证，验证器默认只读，实验产物不能冒充正式交付。

### 4.1 文件管理总原则

1. **先找位置，再创建文件**：创建前必须先确定文件类型、Owner、权威关系和生命周期。
2. **根目录最小化**：仓库根只放项目入口、许可证、依赖/构建入口和工具强制要求的配置。
3. **同类归一**：相同职责的文件进入同一目录，不因执行者、日期或工具不同散落到多个位置。
4. **一个职责一个canonical位置**：同一配置、注册表、规则或脚本不能存在多个可写副本。
5. **临时与正式隔离**：实验、缓存、日志、报告、生成物不得混入正式源码和权威文档目录。
6. **源文件与生成物分离**：生成物必须可以追溯输入和生成器，且不能反向成为不可说明的手工事实源。
7. **不靠文件名猜用途**：删除、归档和迁移必须读取内容、引用和版本关系。
8. **目录也有Owner和准入规则**：不能因为路径存在就允许任意文件进入。

### 4.2 根目录允许清单

项目应维护`config/root_allowlist.json`或等价规则。仓库根通常只允许：

- 项目启动入口：`PROJECT_RULES.md`、`README.md`。
- 法律和协作文件：`LICENSE`、`CONTRIBUTING.md`、`SECURITY.md`。
- 版本控制配置：`.gitignore`、`.gitattributes`。
- 包管理、构建和工作区必须位于根层的入口文件。
- 经过记录的工具专用配置。

下列内容默认不得直接创建在根目录：

- 临时脚本、一次性修复脚本、调试输出。
- 测试结果、截图、日志、扫描报告。
- 业务方案、审计报告、任务卡和交接文件。
- 下载文件、压缩包、模型、数据导出和生成资产。
- 名称为`new`、`final`、`tmp`、`copy`、`backup`的模糊副本。

根目录新增文件属于C2目录治理变更；必须说明为什么既有目录都不适用，并更新允许清单或目录说明。

### 4.3 文件类型到目录的默认映射

| 文件类型 | 推荐位置 | 规则 |
|---|---|---|
| 产品源码 | `src/`或框架约定源码目录 | 按领域/模块组织，不按开发者组织 |
| 测试 | `tests/`或与源码同级的框架约定位置 | 目录结构尽量映射被测模块 |
| 公共接口/类型 | `src/contracts/`、`src/types/`或领域内部 | 禁止建立多个`common/types`垃圾桶 |
| 配置schema与正式配置 | `config/` | schema、默认值、环境差异和迁移分层 |
| 业务文档 | `docs/product/`或`docs/design/` | 通过权威索引路由 |
| 治理规则 | `docs/governance/` | 只保留current入口及明确历史关系 |
| ADR/决策 | `docs/decisions/` | 稳定编号，不随实现文件移动 |
| 任务卡/Change List | `docs/tasks/` | 必须带任务ID |
| 执行报告 | `docs/tasks/reports/` | 不与current设计混放 |
| 多AI声明与交接 | `docs/tasks/coordination/` | 每个任务独立文件 |
| 验证器 | `tools/validators/` | 默认只读，返回明确退出码 |
| 生成器 | `tools/generators/` | 声明输入、输出和确定性 |
| 维护/迁移脚本 | `tools/maintenance/` | 默认dry-run，可重复执行 |
| 一次性实验 | `experimental/<TASK-ID>/` | 有Owner、到期日和转正规则 |
| 临时工作区 | `.scratch/<TASK-ID>/` | 默认gitignore，任务结束清理申请 |
| 缓存 | `.cache/`或工具约定目录 | 不作为事实源，默认gitignore |
| 日志与运行输出 | `outputs/logs/` | 有保留期，不进入源码目录 |
| 派生报告 | `outputs/reports/`或任务报告目录 | 可重建并登记生成命令 |
| 原始资产/数据 | `assets/source/`或`data/raw/` | 只读保存来源和许可 |
| 处理后资产/数据 | `assets/processed/`或`data/processed/` | 记录输入hash和工具版本 |
| 发布产物 | `dist/`、`build/`或`releases/` | 不与源码混放，通常不直接提交 |
| 第三方代码 | `vendor/`或包管理器目录 | 记录来源、版本和许可证，不直接魔改 |
| 废弃内容 | `deprecated/` | 不得被current入口继续引用 |
| 历史归档 | `archive/` | 只读，保留索引和归档原因 |

语言、框架或平台强制的目录约定优先于本表，但必须在目录地图中登记，不得同时保留一套“自定义等价目录”。

### 4.4 创建文件前决策树

任何人工或AI创建文件前必须依次回答：

1. **这个输出必须成为文件吗？** 临时计算结果可只进入日志或任务证据，不必永久保存。
2. **仓库里是否已有同职责文件？** 按名称、关键词、符号、权威索引和内容hash搜索。
3. **应修改canonical文件还是新建？** 只有职责、生命周期或受众确实不同才新建。
4. **文件属于哪种类型？** 源码、测试、配置、文档、报告、生成物、缓存、资产、实验或归档。
5. **对应目录是否已有准入说明？** 有则遵守；没有则先补目录Owner和用途。
6. **谁负责维护？何时失效？** 无Owner、无维护触发器的长期文件不得创建。
7. **需要被索引或注册吗？** current文档、任务、配置、资产、Skill、插件和生成物必须进入对应注册表。
8. **会不会与其他AI的plannedNewFiles冲突？** 新路径也必须声明，不能只锁已有文件。
9. **命名能否长期稳定？** 禁止用`final_v2_new`等状态型文件名代替版本管理。
10. **验证和回滚是什么？** 新文件必须有存在性、引用、格式及业务验证；错误创建应能安全撤回。

任一问题无法回答时，将文件保持在任务级`.scratch/<TASK-ID>/`，不得直接进入正式目录。

### 4.5 新文件创建申请最小字段

Change List中的每个NEW文件至少写明：

```text
Path:
Type:
Purpose:
Canonical For:
Owner:
Authoritative Source:
Consumers/References:
Generated or Handwritten:
Retention/Lifecycle:
Index/Registry Update:
Validation:
Rollback:
```

只写“新增文档”“新增脚本”不足以获得创建授权。

### 4.6 新目录创建规则

新增目录比新增文件影响更大，因为它会形成长期分类。创建前必须证明：

- 现有目录不能表达该职责。
- 预计包含至少一个明确文件族，而不是只为单个文件套一层目录。
- 与现有目录名称和职责不重叠。
- 有Owner、允许/禁止文件类型、命名规则和清理触发器。
- 已决定是否需要README、索引、注册表、gitignore和自动门禁。

建议每个非显而易见的长期目录提供`README.md`，至少说明：

```text
Purpose
Owner
Allowed Contents
Forbidden Contents
Canonical Sources
Generated Contents
Retention/Cleanup
Validation
```

禁止创建`misc/`、`others/`、`temp/`、`new/`、`final/`等无稳定职责的长期目录。

### 4.7 文件命名规则

- 优先遵守语言和框架约定；同一目录只采用一种主命名风格。
- 名称表达职责或领域，不表达作者、情绪和临时状态。
- 禁止用`最终版`、`最新版`、`new`、`copy`、`backup2`管理版本；使用Git、版本字段或日期化快照目录。
- 任务产物必须含稳定任务ID，例如`TASK-042_CHANGE_LIST.md`。
- 报告可带日期，但current入口不应靠日期文件名不断复制；应更新current并归档旧快照。
- 文件扩展名必须与真实格式一致。
- 大小写规则必须跨平台安全；禁止仅大小写不同的两个路径。
- 避免控制字符、尾随空格、保留设备名和不同系统不兼容字符。
- 重命名属于破坏性操作，不能通过“新建同内容文件”绕过迁移安全门。

### 4.8 防重复与canonical判定

创建前至少执行：

- 路径和近似文件名搜索。
- 标题、关键术语、接口名或配置键搜索。
- 内容hash搜索，发现完全相同副本时不再创建。
- 权威索引、注册表和引用关系检查。
- 生成器输出目标检查。

发现重复候选时按以下顺序处理：

1. 确认canonical来源和消费者。
2. 判断副本是镜像、缓存、快照、兼容层还是误复制。
3. 先停止新增，不立即删除任何一份。
4. 需要合并时建立独立迁移任务、引用清单和回滚点。
5. 合并完成并重扫后，才申请归档或删除非canonical副本。

不得用mtime、新旧文件名或路径深浅自动决定canonical。

### 4.9 源文件、生成物和派生报告

每类生成物必须定义：

- 唯一输入事实源。
- 生成器路径和版本/hash。
- 输出目录和允许文件模式。
- 是否提交版本控制。
- 是否允许手工编辑。
- 重建命令和确定性预期。
- 清理和保留策略。

生成目录应包含manifest或在注册表登记：`input_sha256`、`generator_sha256`、`generated_at`、`output_sha256`。如果生成物允许手工修订，必须明确哪个方向是canonical，禁止双向无规则同步。

验证器生成的报告属于派生证据。重复运行刷新报告不应被误判为业务修改，但必须在任务报告中如实登记。

### 4.10 临时文件、缓存、日志和实验

- 临时文件只能进入`.scratch/<TASK-ID>/`或系统临时目录。
- `.scratch/`、缓存、构建中间物和本地日志默认加入gitignore。
- 临时文件必须有Owner和到期时间；任务关闭前决定删除、转正或归档。
- 实验代码进入`experimental/<TASK-ID>/`，不得被生产入口默认引用。
- 实验转正必须重新经过正常代码审查、测试、目录放置和注册流程，不能直接移动后宣称完成。
- 调试脚本不得混入生产源码；仍需长期使用时转为有测试和文档的正式工具。
- 日志不得包含密钥、个人数据或完整敏感载荷；必须定义保留期和脱敏规则。

### 4.11 文件生命周期状态

建议对重要文件或注册表条目使用：

| 状态 | 含义 | 允许行为 |
|---|---|---|
| proposed | 已提议，尚未进入正式目录 | 只在任务或scratch中 |
| active | 当前维护和使用 | 可按Owner规则修改 |
| frozen | 发布/审查/迁移窗口冻结 | 只读，变化使证据失效 |
| superseded | 已被明确文件替代 | 只读，保留替代链接 |
| deprecated | 不再用于新工作 | 只读或受限维护 |
| archived | 历史保存 | 默认只读 |
| deletion_candidate | 完成只读清理评估 | 仍不得删除 |
| deleted | 已经P0安全门执行 | 必须有回执和恢复点 |

状态变化必须有任务ID、原因、Owner、日期和替代/恢复路径。

### 4.12 移动、重命名与归档

移动和重命名会破坏引用、构建、导入、链接、注册表和外部自动化，按C2/C3处理：

1. 先输出old→new映射和dry-run引用清单。
2. 冻结全部相关文件SHA。
3. 检查代码导入、文档链接、构建脚本、CI、配置、注册表和外部消费者。
4. 建立Git安全点或等价可恢复点。
5. 原子执行路径变化和引用更新，不保留无说明双入口。
6. 运行路径、链接、构建和全量测试。
7. 更新目录README、索引、注册表和handoff。

归档不是垃圾桶。归档目录必须有索引，记录原路径、归档原因、日期、替代入口、恢复方法和保留期限。

### 4.13 文件索引与目录地图

项目应维护机器可读的`config/file_placement_rules.json`或等价配置，至少包含：

```json
{
  "rootAllowlist": [],
  "directoryRules": [
    {
      "path": "docs/tasks/reports",
      "owner": "project-governance",
      "allowedPatterns": ["*.md"],
      "forbiddenPatterns": ["*.bin", "*.zip"],
      "generated": false,
      "retention": "project-lifetime"
    }
  ]
}
```

大型项目可增加`FILE_CATALOG.json`，登记关键文件的stable ID、type、canonical状态、Owner、消费者、生成来源和生命周期；普通源码不必逐文件人工登记，避免目录治理本身失控。

### 4.14 自动放置门禁

文件治理检查器至少阻断：

- 根目录出现未登记文件。
- 正式目录出现临时、备份、日志、压缩包或缓存模式。
- 文件扩展名与允许类型不符。
- 同路径仅大小写不同。
- current/注册表/关键配置出现多个canonical副本。
- 生成物出现在源码目录或缺少生成登记。
- 实验/废弃文件被生产入口引用。
- 新目录缺Owner或用途说明。
- 任务创建了未在`plannedNewFiles`声明的路径。
- 任务结束仍遗留未登记scratch文件。
- 绝对路径、U+FFFD、秘密形态或坏链接进入正式文本。

门禁先以只读报告模式建立基线，再对新增违规强制失败；不得为了让旧仓库一次变绿而直接批量移动文件。

### 4.15 定期维护

建议每周或每个里程碑执行：

1. 根目录允许清单差异。
2. 未索引current文档和孤立任务报告。
3. 重复hash、近似命名和canonical冲突。
4. 过期scratch、日志、缓存和实验。
5. 生成物与输入/生成器hash漂移。
6. 废弃内容仍被current或生产入口引用。
7. 空目录、无Owner目录和职责重叠目录。
8. 大文件、压缩包、二进制和敏感数据异常增长。

扫描只产生候选清单，不自动删除或移动。所有清理仍走独立Change List和破坏性操作安全门。

## 5. 角色与责任

| 角色 | 核心责任 | 禁止行为 |
|---|---|---|
| Decision Owner | 业务取舍、范围、优先级、最终验收 | 把互相冲突的决定同时标current |
| Coordinator | 拆分任务、检查冲突、分配角色 | 直接覆盖执行者未交接文件 |
| Executor | 在冻结范围内实施和局部验证 | 扩大范围、自行标accepted |
| Reviewer | 独立检查设计、实现、测试和风险 | 直接修改被审文件后仍称独立审查 |
| Integrator | 串行集成共享文件、冻结验证窗口 | 在其他写入未冻结时做最终测试 |
| Curator | 管理长期记忆、索引、注册表 | 把临时对话内容直接升级为长期事实 |
| Release Owner | 发布批准、回滚和外部传播 | 用局部测试替代发布门禁 |

小团队可以一人承担多个角色，但同一高风险变更的Executor和独立Reviewer不得是同一执行主体。

### 5.1 RACI分配

每个C2/C3任务应明确：

| 字段 | 说明 |
|---|---|
| Responsible | 实际执行者，可多个但文件不重叠 |
| Accountable | 对结果最终负责，只能一个 |
| Consulted | 必须在执行前或审查时咨询 |
| Informed | 完成、阻断或事故时通知 |

没有Accountable的任务不得进入in_progress；多个Accountable必须先指定唯一裁决者。

### 5.2 角色指派与交接

- 角色指派记录任务ID、主体、有效期、权限和替代者。
- AI会话结束不代表角色自动释放；必须写handoff或由Coordinator明确取消。
- Reviewer不可使用Executor未披露的“私下上下文”作审查依据，审查只基于冻结文件和证据。
- Integrator更换时必须交接活动队列、热文件SHA、冻结状态和未完成验证。
- Release Owner缺席时默认不发布，不能由Executor自动继任。

### 5.3 职责缺口处理

若项目没有独立Reviewer、Curator或Integrator：

1. 把缺口标为风险而不是假装角色存在。
2. C0/C1可按规则降级为自检。
3. C2可由独立时间窗口复审，但需明确“非独立主体”限制。
4. C3必须等待外部复核或Decision Owner书面接受风险。

### 5.4 角色行为门禁

自动检查应阻断：Executor将任务标integrated/accepted、Reviewer直接修改claimed文件、多个Integrator同时active、热文件被普通Executor声明、无Owner任务进入in_progress。

## 6. 优先级、风险和证据模型

### 6.1 优先级

| 等级 | 定义 |
|---|---|
| P0 | 安全、数据损坏、授权越界、发布事故、不可逆操作 |
| P1 | 核心功能、公共契约、确定性、重大回归 |
| P2 | 一般缺陷、可维护性、局部体验 |
| P3 | 建议、优化、非阻断改进 |

### 6.2 变更风险

| 等级 | 示例 | 管理强度 |
|---|---|---|
| C0 | 注释、纯文案、只读报告 | 自检 |
| C1 | 单文件局部修改、无公共契约 | 目标测试 |
| C2 | 多文件、公共接口、配置、数据、自动化 | 影响图、独立审查、全量回归 |
| C3 | 架构、依赖、构建、发布、安全、迁移、破坏性操作 | 冻结授权、独立审查、回滚方案、最终冻结验证 |

### 6.3 证据等级

| 等级 | 证据 |
|---|---|
| E0 | 推测或未验证描述 |
| E1 | 文件存在、静态读取 |
| E2 | 人工检查、结构对照 |
| E3 | 自动检查、单元/集成测试、可复现命令 |
| E4 | 真实环境、编辑器、设备、生产或人工业务验收 |

验证声明必须写明证据等级。需要E4的事项不能由E3替代。

### 6.4 风险评分

除C0-C3外，可用五维评分辅助判断：影响范围、可逆性、数据敏感度、外部依赖、验证难度，每项0-3分。

| 总分 | 默认等级 |
|---:|---|
| 0-3 | C0 |
| 4-6 | C1 |
| 7-10 | C2 |
| 11-15 | C3 |

任何P0、安全、生产数据、历史重写和不可逆发布直接提升为C3，不因总分低而降级。

### 6.5 证据新鲜度

证据必须绑定：输入SHA、工具版本、环境、执行时间、退出码和日志位置。出现以下任一情况，旧证据失效：

- 受测文件或依赖变化。
- 规则/测试/配置变化。
- 构建环境、平台或工具主版本变化。
- 超过领域规定的有效期。
- 发现证据生成过程有缺陷。

### 6.6 风险登记

风险记录字段：`Risk ID / Description / Probability / Impact / Level / Trigger / Owner / Mitigation / Contingency / Due / Status / Evidence`。

风险不等于问题：风险尚未发生，问题已经发生。问题必须进入缺陷/RCA或阻断流程，不能长期留在风险表中规避处理。

### 6.7 风险升级

- P0立即通知Decision Owner并冻结相关外部传播/发布。
- P1在当前任务关闭前修复或取得明确风险接受。
- P2进入有Owner和触发器的backlog。
- P3可记录建议，但不得写成已实施。

## 7. 需求、路线图和进度管理

### 7.1 七值进度状态

只允许以下状态：

| 状态 | 含义 |
|---|---|
| unassessed | 未取得足够证据，不能判断 |
| not_started | 已确认任务但尚未开始 |
| in_progress | 有实际执行证据且未完成 |
| blocked | 有明确阻断和解除条件 |
| verification_pending | 实施完成，等待验证/验收 |
| verified | 已按规定证据验证 |
| deprecated | 不再排期，仅保留历史/兼容参考 |

`completed`不作为进度文件状态；“完成”容易混淆实施完成、验证完成和用户接受。最终接受使用独立的`accepted`验收记录。

### 7.2 进度文件最小字段

```text
Status:
Owner:
Scope:
Evidence:
Last Verified:
Not Completed Items:
Block Reason:
Unblock Condition:
Next Review Trigger:
```

进度索引必须由个体进度文件生成或校验，不能手工维护两套相互独立的状态。

### 7.3 路线图规则

- 路线图只安排已裁定目标，不替代需求、架构或验收文档。
- 每项路线图任务必须有Owner、依赖、状态、退出条件和证据入口。
- 外部日期变化不等于任务自动延期；必须记录决策。
- blocked必须同时写阻断原因和解除条件。

### 7.4 需求进入流程

每个需求必须经过：提出→澄清→去重→价值/风险评估→验收定义→依赖分析→优先级裁定→进入路线图。

需求卡至少包含：用户/业务问题、期望结果、非目标、验收场景、约束、依赖、风险、Owner、证据和失效条件。只有“实现某功能”而没有问题和验收的需求保持unassessed。

### 7.5 Backlog治理

- 每项backlog有稳定ID，不以文件行号作为身份。
- 重复需求合并到canonical项，保留来源关系。
- 超过复查期限且无Owner的项标stale，不自动删除。
- 已被战略决定取消的项标deprecated并说明原因。
- 技术债与业务需求分开计数，但可通过依赖关联。
- 不允许用一个“其他优化”任务容纳无限范围。

### 7.6 依赖图

任务依赖必须区分：硬依赖、软依赖、外部依赖、验证依赖。循环硬依赖必须先拆分或形成原子任务，不能让两个任务互相等待。

依赖变化时同步路线图、任务状态和阻断条件。仅在依赖的验收证据完成后，后续任务才可解除blocked。

### 7.7 状态迁移规则

```text
unassessed → not_started → in_progress → verification_pending → verified
                     ↘ blocked ↗
任意非终态 → deprecated
```

- 没有实际写入/执行证据不能进入in_progress。
- 没有完整实施回执不能进入verification_pending。
- 没有规定证据不能进入verified。
- verified后的目标或输入变化必须重新评估，不能永久保持绿色。

### 7.8 变更控制

执行中新增需求先进入“变更请求”，评估是否属于原目标：

- 不改变范围和验收的澄清：更新任务说明并记录。
- 增加文件、接口、依赖或风险：冻结当前任务，修订Change List并重新授权。
- 独立目标：建立新任务，不塞入当前批次。
- 紧急P0：停止普通任务，按事故流程处理。

## 8. 标准任务生命周期

所有写入任务采用：

```text
扫描 → 任务卡 → 影响分析 → 冻结范围/SHA → 冲突检查
→ 执行 → 局部验证 → 独立审查 → 冻结写入
→ 全量验证 → 报告/Handoff → 用户或授权者验收
```

### 8.1 任务开始条件

- 目标和非目标明确。
- 已找到适用current文档。
- 文件范围可精确列举。
- 多AI声明无冲突。
- 执行前SHA已记录。
- 风险等级和验证计划明确。
- 删除、安装、外部写入等特殊权限已单列。

### 8.2 写入规则

- 每个文件写入前重新计算SHA。
- SHA变化则停止，不覆盖、不回滚、不自动合并。
- 每次只修改任务卡允许的内容。
- 发现范围外问题，记录到报告或新任务候选，不顺手修复。
- 生成物必须记录生成器版本、输入和可重复命令。

### 8.3 完成条件

- 实际Change List与冻结清单一致。
- 目标文件前后SHA可追溯。
- 局部测试和规定门禁已执行。
- 新增失败=0。
- C2/C3的P0/P1审查意见=0。
- 最终测试绑定的文件SHA在测试后未变化。
- 报告明确列出未解决项。
- 只有用户或授权验收者可标`accepted`。

### 8.4 执行模式

| 模式 | 允许动作 | 适用情况 |
|---|---|---|
| Read-Only Review | 读取、扫描、测试、报告；不改目标 | 审计、诊断、验收 |
| Normal Execution | 输出计划后等待一次授权 | 范围尚需用户确认 |
| Direct Execution | 用户已明确授权清晰范围，回执后直接执行 | 常规构建/修改 |
| Emergency Containment | 只执行止血和证据保全 | P0事故；不自动扩大到根因重构 |

模式不改变删除、发布、外部写入、依赖安装和历史重写所需的独立授权。

### 8.5 影响分析

写入前至少检查：直接消费者、间接消费者、公共契约、配置/数据、自动化、文档、进度、记忆、Skill、发布和回滚。影响未知项必须写unassessed，不能省略。

### 8.6 失败与重试

- 同一命令失败后先保存输出并分类：环境、权限、输入、工具、实现或外部服务。
- 不得在没有新证据的情况下反复运行相同失败命令。
- 重试必须记录变化的前提。
- 工具不可用时使用安全替代检查，但状态保持NOT_RUN/NOT_IMPLEMENTED。
- 连续三次出现相同阻断且无新路径时升级Coordinator/用户，不通过扩大范围“试试看”。

### 8.7 回滚

回滚方案必须说明触发条件、可恢复点、回滚范围、数据兼容、验证和Owner。回滚不是默认使用`git reset --hard`；应优先用任务级反向变更、备份恢复或版本切换，并保护其他未提交工作。

### 8.8 验收分离

- implemented：代码/文档已写入。
- verification_pending：执行者完成规定自检。
- verified：独立证据满足技术条件。
- accepted：Decision Owner确认业务结果。
- integrated：Integrator完成共享集成。
- released：Release Owner完成外部发布。

这些状态不能互相替代。一个任务可以verified但未accepted，也可以accepted但尚未released。

## 9. 防上下文超限与任务漂移

对话上下文不是任务事实源。每个长任务建立四个持久文件：

1. `CHANGE_LIST.md`：不可变目标、范围、非目标、基线SHA、验收条件。
2. `TASK_STATE.json`：当前步骤、完成项、待办、阻断、下一条唯一动作。
3. `EVIDENCE.md`：命令、结果、日志路径、测试和SHA。
4. `HANDOFF.md`：换对话、换执行者或上下文压缩后的恢复入口。

### 9.1 强制检查点

以下时机必须更新checkpoint：

- 每完成一个文件。
- 每完成一个验证步骤。
- 执行长命令前后。
- 连续工作约15分钟。
- 用户插入新要求。
- 发现上下文压缩、摘要或接近上限。
- 结束当前对话前。

检查点至少记录：任务ID、Change List SHA、当前步骤、下一动作、已改文件及SHA、未改文件、验证结果、已知失败、HEAD和禁止范围。

### 9.2 恢复门禁

换对话或上下文压缩后，执行者必须重新读取：项目入口、Change List、TASK_STATE、最近checkpoint/Handoff、活动任务和目标文件SHA。

只有以下条件全部满足才能继续：

- 冻结锁一致。
- 目标文件SHA一致。
- 当前磁盘变更能由任务状态解释。
- 没有新的文件占用冲突。
- 下一条动作唯一且仍在范围内。

聊天摘要与磁盘事实冲突时，以冻结文件和实际SHA为准。

### 9.3 批次控制

- 文档批次建议不超过5个目标文件。
- 代码批次只解决一个共同根因。
- 大规模任务使用总控调度多个独立小批次。
- 每批独立验证后再启动下一批。

### 9.4 上下文预算分级

AI平台不一定提供精确token余量，因此同时使用平台信号和工作量信号：

| 区间 | 信号 | 行为 |
|---|---|---|
| Green | 任务刚开始、范围清晰、无压缩提示 | 正常执行，按文件checkpoint |
| Yellow | 已处理多个文件、输出很长、出现摘要/裁剪风险 | 停止扩展范围，更新TASK_STATE和EVIDENCE |
| Red | 平台明确压缩、关键早期内容不可见、无法复述冻结锁 | 不再写目标文件，先handoff并执行恢复门禁 |

进入Yellow后不得启动新的子批次；进入Red后即使“似乎记得”也不能继续写入。

### 9.5 TASK_STATE写入规则

- 机器状态使用JSON/YAML等结构化格式，叙述证据放EVIDENCE/HANDOFF。
- `nextAction`必须只有一个可执行动作，禁止“继续完成剩余内容”。
- `completedFiles`包含最终SHA和验证状态，不只写文件名。
- 状态文件不修改Change List目标、验收和授权。
- 状态更新使用完整重写或原子替换，避免半写入；写后重新解析验证。
- TASK_STATE自身不写自身最终SHA，最终SHA放checkpoint或外部回执。

### 9.6 恢复证明

恢复后先输出：

```text
Task ID
Change List SHA
Checkpoint/Handoff SHA
Last completed step
Observed modified files
Baseline match
Active-task conflict result
Single next action
Resume decision: ALLOWED / STOPPED
```

只有`ALLOWED`才能继续。`STOPPED`必须说明不一致对象和需要的裁决，不得用新摘要覆盖旧状态。

### 9.7 幂等与重复执行

每个步骤应尽量设计为幂等：重复运行不会重复插入、重复创建或破坏状态。不能幂等的步骤必须有执行标记、前置检测和恢复方案。

恢复时先检测步骤是否已经部分执行：

- 目标文件是否已有预期片段。
- 生成物是否存在且输入hash一致。
- 外部操作是否已有请求ID/回执。
- 数据迁移是否有版本标记。

禁止仅根据TASK_STATE写“未完成”就盲目重复外部操作。

### 9.8 上下文恢复演练

C3长任务在正式执行前应做一次恢复演练：由新会话只读读取四份任务文件，输出恢复证明和下一动作。无法无歧义恢复时，说明任务文件不完整，必须先修复治理材料。

## 10. 多 AI 协作

### 10.1 协调控制面

每个活动任务建立唯一文件：

```text
coordination/active/TASK_<ID>.json
coordination/baselines/BASELINE_<ID>.json
coordination/checkpoints/CHECKPOINT_<ID>.json
coordination/handoffs/HANDOFF_<ID>.md
coordination/findings/REVIEW_<ID>.md
```

任务声明至少包含：owner、role、status、claimedFiles、plannedNewFiles、integrationOnlyFiles、baseline、lease时间和handoff路径。

### 10.2 并行规则

- 路径不重叠的C0-C2任务可以并行。
- 同一文件只能有一个Writer。
- 总索引、构建文件、共享配置、注册表和发布文件属于热文件，只能由Integrator串行修改。
- 租约超时只表示疑似停滞，不能自动抢占。
- 不允许多个AI分别生成同一路径，再尝试“择优覆盖”。

### 10.3 交接规则

Executor完成局部验证后最多标`ready_for_integration`。Integrator冻结所有写入，串行处理共享变更，再运行全量验证。只有Integrator可标`integrated`，只有授权验收者可标`accepted`。

### 10.4 文件分级

| 等级 | 示例 | 写权限 |
|---|---|---|
| G0 | 项目规则、权威索引、总体进度、发布政策 | Decision Owner/Integrator串行 |
| G1 | 构建入口、依赖锁、共享schema、注册表、CI | Integrator或独立C3任务 |
| G2 | 单模块源码、测试、局部文档 | 不重叠时可并行 |
| G3 | 带任务ID的报告、checkpoint、handoff | 对应任务独占 |

Executor需要G0/G1变化时，只写`integrationOnlyFiles`和handoff请求，不直接声明为普通claimedFiles。

### 10.5 租约状态机

```text
draft → requested → active → ready_for_integration → integrated
                  ↘ blocked
                  ↘ cancelled
```

- requested只有Coordinator/用户批准后变active。
- 租约包含开始、预期结束、最后heartbeat和Owner。
- 超时不能自动释放；Coordinator需检查handoff、进程和磁盘变化。
- cancelled必须说明目标文件是否有未完成修改及如何处置。
- integrated后租约关闭，但历史记录保留。

### 10.6 冲突矩阵

| 冲突 | 处理 |
|---|---|
| claimedFiles完全重叠 | 后申请者blocked |
| plannedNewFiles路径相同 | 两者都停止，Coordinator裁定canonical任务 |
| 一个任务修改、另一个任务验证同一文件 | 先完成写入并冻结，再启动验证 |
| G0/G1并发请求 | 进入Integrator队列，串行处理 |
| baseline SHA漂移 | 当前任务blocked，不覆盖 |
| 两个任务产生不同接口方案 | 升级为权威/ADR裁决，不在代码层合并 |
| 两个Reviewer结论冲突 | 由Accountable基于证据裁决并记录 |

### 10.7 集成队列

每个集成请求记录：request ID、来源任务、目标热文件、所需修改、依赖、执行前SHA、冲突请求、验证和回滚。Integrator按依赖和优先级串行执行，不按AI完成时间先到先得覆盖。

集成前要求来源任务ready、handoff完整、局部测试完成、P0/P1审查意见关闭。集成后重新运行共享门禁，来源任务的局部PASS不能代替集成验证。

### 10.8 分支与工作区

- 能使用独立worktree/branch时，每个高风险任务使用独立工作区，减少未提交变化污染。
- 共享同一工作区时，必须使用文件声明和写前SHA，不依赖分支名隔离。
- 不允许一个AI在未交接时切换或重置其他AI的分支。
- 合并冲突不得由模型根据“更完整”自动选择；必须回到权威和契约裁决。

### 10.9 协作机器门禁

至少检查：任务ID唯一、角色合法、claimed/planned路径不重叠、G0/G1权限、baseline存在、SHA漂移、租约状态、handoff必需字段、Reviewer与Executor隔离、integrated权限和过期租约。

检查器默认只读，不自动释放租约、合并文件或修改状态。

### 10.10 最终验证窗口

Integrator宣布冻结后：

1. 记录全部受测文件和依赖SHA。
2. 确认没有active Writer。
3. 串行运行门禁和测试。
4. 记录环境、命令、退出码和日志。
5. 重新核对受测SHA未变化。
6. 写最终集成回执。

测试期间任何受测文件变化都会使结果失效，必须重新冻结和重跑。

## 11. 文档管理

### 11.1 文档类型

| 类型 | 用途 |
|---|---|
| current | 当前有效规则或事实入口 |
| target | 目标设计，不代表已实施 |
| migration | 当前到目标的迁移步骤 |
| supporting | 论证、分析、证据 |
| report | 某次执行和验证结果 |
| historical | 历史快照，不指导当前执行 |
| deprecated | 已废弃，不得作为新任务依据 |

### 11.2 防漂移规则

- 同一领域最多一个current入口。
- current变化必须同步权威索引。
- 被替代文档写明`superseded_by`，不通过删除掩盖历史。
- 文档链接使用相对路径并自动检查存在性。
- 示例占位符不得残留在正式current区段。
- 报告不能提升自身状态为accepted。

### 11.3 文档门禁

至少自动检查：唯一current、合法状态、必需元数据、相对路径、链接有效、替代关系、占位符、平台/产品口径冲突和进度计数一致性。

### 11.4 文档创建流程

创建前先确认文档类型、领域、目标读者、Owner、是否已有canonical入口、维护触发器和退出路径。NEW文档必须在Change List中声明，并说明是否同步权威索引。

禁止因为旧文档难改就创建“新版”“最终版”绕开冲突。若需要重构，应建立迁移任务和supersedes链。

### 11.5 文档审查维度

1. **权威**：来源和优先级是否正确。
2. **事实**：当前、目标、迁移、推测是否分离。
3. **范围**：适用平台、版本、模块和日期是否明确。
4. **一致性**：术语、状态、接口、路径和数量是否与事实源一致。
5. **可执行性**：执行者能否得到精确输入、动作、停止条件和验收。
6. **可维护性**：Owner、触发器和替代关系是否存在。
7. **安全**：是否包含密钥、隐私、机器路径或不应传播的信息。

### 11.6 文档状态迁移

```text
draft → review_pending → current
current → superseded / deprecated
supporting → historical
report → verification_pending → verified → accepted（外部验收记录）
```

文档内容变化不自动提升实施状态。设计文档current只说明设计有效，不说明产品已经实现。

### 11.7 快照与历史

- current文档优先原地维护并依赖Git历史，避免日期副本泛滥。
- 法规、发布、审计等确需快照时放入明确快照目录并登记来源current SHA。
- historical文档顶部必须明确“不得指导当前执行”和当前替代入口。
- 归档/历史文档默认只读，修正事实错误需保留更正说明。

### 11.8 图表、表格和代码示例

- 图表必须有文本事实源，不能成为唯一不可检索决定。
- 表格列数和状态值自动检查。
- 示例代码标明伪代码/可运行代码、版本和适用范围。
- 示例占位符不得被复制进生产配置。
- 外部链接应有标题、访问日期和必要的内部摘要；关键规则不能只依赖易失外链。

### 11.9 文档维护触发器

接口、平台、默认配置、发布流程、状态词、目录结构、Owner、测试命令或权威关系变化时，自动产生文档同步任务。代码变更报告必须列出“文档影响：yes/no及依据”。

## 12. 项目记忆管理

### 12.1 记忆分层

| 层级 | 内容 | 写权限 |
|---|---|---|
| G0索引 | 稳定规则入口和主题路由 | Curator |
| G1主题 | 长期有效决定和约束 | Curator合并 |
| G2 Inbox | 带任务ID的候选补丁 | Executor可新增 |
| G3日志 | 临时执行记录 | 对应任务 |
| Archive | 失效/历史内容 | Curator |

项目只能有一个活动记忆根。迁移副本、工具缓存、个人记忆和自动化局部记忆不得与活动根竞争。

### 12.2 记忆准入

每条长期记忆必须包含：稳定ID、主题、事实/决策类型、来源、证据、Owner、创建时间、复查时间、失效条件和敏感级别。

禁止写入：密钥、个人隐私、无来源推测、实时任务进度、机器绝对路径、已过期平台口径和大段聊天记录。

### 12.3 合并与恢复

- Executor只提交Inbox补丁。
- 唯一Curator在冻结窗口执行去重、冲突检查、脱敏和容量检查。
- 同ID不同内容必须人工裁决。
- 合并前后记录SHA和恢复点。
- 过期条目应标失效/归档，不直接抹除决策历史。

### 12.4 唯一活动根判定

盘点所有候选记忆目录，记录文件数、主索引、主题、inbox、更新时间、调用方和内容hash。按显式项目规则、工具实际读取入口和历史证据裁定唯一活动根。

不能通过目录更新时间、文件数量或名称相似自动选择。其他根必须标为legacy、migration-template、task-local、external或archive，并禁止双向同步。

### 12.5 记忆条目schema

```text
id:
topic:
kind: fact / decision / constraint / preference / lesson
statement:
source:
evidence_level:
owner:
created_at:
last_validated:
review_after:
invalidated_by:
sensitivity:
supersedes:
```

statement应短小、原子化；一个条目只表达一个可验证事实或决定。大型说明放权威文档，记忆只保存路由和结论。

### 12.6 启动读取协议

AI启动时先读取记忆索引，再按任务主题加载必要条目，不一次加载全部历史。读取后必须与current权威文档和磁盘事实交叉检查；记忆冲突时以更高权威为准并提交修正补丁。

### 12.7 Inbox状态机

```text
proposed → validated → merged
        ↘ rejected
        ↘ conflict
        ↘ expired
```

补丁必须有task ID、目标topic、来源、证据、敏感级别和期望操作。Curator不得合并缺字段、重复ID、秘密命中或与current冲突的补丁。

### 12.8 冲突处理

| 情况 | 动作 |
|---|---|
| 同ID同内容 | 去重，保留来源 |
| 同ID不同内容 | conflict，等待裁决 |
| 不同ID同事实 | 选择canonical ID并建立supersedes |
| 记忆与current冲突 | 不覆盖current，提交记忆修正 |
| 记忆与磁盘事实冲突 | 标stale，重新验证来源 |
| 多个AI同时提同主题补丁 | 保持独立patch，由Curator串行合并 |

### 12.9 新鲜度与容量

- 条目按领域定义review_after；到期不是自动删除，而是停止当作已验证事实。
- topic文件设置大小/条目阈值，超过后拆主题或归档旧条目。
- 日志按日期和保留期轮转，不把执行日志长期注入上下文。
- 索引只保留稳定路由，不复制全文。

### 12.10 安全与隐私

- 合并前运行脱敏秘密扫描，输出只含规则ID、路径、行号和不可逆指纹。
- restricted内容不得进入普通项目记忆。
- 外部用户偏好与项目事实分开。
- 个人信息按最小化、目的限定和保留期管理。
- 疑似秘密命中立即隔离补丁，不在聊天复述字面值。

### 12.11 备份、迁移和恢复

- 合并前保存活动根manifest和SHA。
- 备份只读、带日期/版本和恢复说明，不与活动根竞争。
- 迁移先复制验证，再切换唯一读取入口，最后将旧根标legacy；不自动删除旧根。
- 恢复后重新运行重复ID、索引、秘密、容量和权威冲突检查。

### 12.12 记忆机器门禁

阻断：多个活动根、缺schema、重复/冲突ID、过期active事实、超容量、秘密形态、绝对机器路径、旧平台口径、无任务ID补丁、非Curator直接修改G0/G1、合并期间并发写入。

## 13. Skill、插件和自动化工具治理

### 13.1 注册表

所有Skill/插件/自动化能力登记：

```text
id, name, provider, status, availability, version,
source_path, installed_targets, content_sha256,
triggers, exclusions, permissions, dependencies,
owner, last_validated, review_after, risk_notes
```

状态建议：`candidate / approved / active / deprecated / quarantined / retired`。

### 13.2 使用规则

- 只有`active + available + hash匹配`的能力可自动调用。
- 来源和安装副本漂移时标`quarantined`，不得按mtime选赢家。
- Skill不能扩大任务授权。
- 读、写、执行、网络、凭据和外部系统权限必须显式声明。
- 触发矩阵至少有正向、负向、冲突优先级、fallback和stop条件。
- 安装、升级、停用、回滚都作为独立任务并更新锁。

### 13.3 信任边界

| 来源 | 默认信任 | 处理 |
|---|---|---|
| 项目版本化Skill | candidate | 审查后可approved/active |
| 用户级安装 | untrusted-until-locked | 比对来源与安装hash |
| 平台内置 | platform-managed | 登记版本/可用性，不由项目修改 |
| 第三方插件/远程服务 | external | 审查权限、数据流、许可和停机降级 |
| 临时生成Skill | candidate | 不得直接进入active |

来源可信不代表当前版本可信；active必须绑定精确内容hash。

### 13.4 选择与加载

1. 根据用户任务匹配触发矩阵。
2. 检查status、availability、provider和安装锁。
3. 完整读取Skill入口及任务必需的直接引用。
4. 声明将采取的额外动作或暂停条件。
5. 只授予任务需要的最小权限。
6. 执行后记录Skill ID、版本/hash和影响。

多个Skill同时匹配时按专用性、显式用户指定、项目优先级和冲突矩阵裁决。无法裁决则不自动组合。

### 13.5 新建Skill准入

- 有明确重复工作和稳定工作流，不是一次性提示集合。
- 名称、描述和触发条件清晰。
- 主入口简洁，详细材料渐进披露。
- 脚本确定、可测试，输出路径和权限明确。
- 至少三条正向、三条负向触发样例。
- 有安全审查、依赖清单、版本、Owner和回滚版本。
- candidate验证通过后才可approved，实际安装锁匹配后才可active。

### 13.6 更新与安装

1. 冻结当前来源和安装hash。
2. 输出变更、兼容、权限和依赖差异。
3. 在隔离位置验证新版本。
4. 运行触发、脚本、安全和回归测试。
5. 更新来源版本、注册表和安装锁。
6. 原子切换active版本。
7. 保留可恢复的rollbackVersion。

禁止直接在安装副本中热修后忘记同步来源；也禁止自动用来源覆盖用户安装中的未知变化。

### 13.7 权限与安全审查

审查：读取范围、写入范围、可执行程序、网络域名、凭据、外部系统、数据上传、递归删除、shell构造、依赖安装和供应链来源。

高风险权限必须在每次任务再次取得授权，不能仅凭Skill active状态长期继承。

### 13.8 降级与停止

- Skill不可用：使用已批准的手工流程或报告NOT_AVAILABLE。
- 安装hash漂移：quarantined并停止调用。
- 输出验证失败：保留证据，不自动换另一个工具反复尝试。
- 外部服务故障：不得绕过安全边界或上传到未批准服务。
- Skill与项目规则冲突：项目规则优先，并登记Skill缺陷。

### 13.9 停用和退休

deprecated表示停止新触发但可能维护兼容；quarantined表示风险未裁决、立即停止；retired表示不再支持。停用前扫描current文档、任务模板、自动化和记忆中的调用引用，更新替代方案。

物理删除Skill仍需文件删除安全门，状态停用不等于授权删除。

### 13.10 Skill机器门禁

检查：ID唯一、同provider/name单active、frontmatter、来源/安装hash、active可用性、缓存/压缩包、权限声明、触发样例、deprecated引用、依赖存在、安全模式和Markdown视图与机器注册表一致。

## 14. 工程实现与代码审查

### 14.1 实现前契约

C2/C3工程任务必须先明确：

- 问题和最小复现。
- 接口/类型定义方。
- 唯一实现。
- 依赖注册或composition root。
- 全部消费方。
- 测试fixture和负向测试。
- 配置、数据、存档、场景、API兼容影响。
- 性能、安全、确定性和平台影响。

公共契约变更必须原子更新定义、实现、注册、消费方和测试，不能留下双路径。

### 14.2 架构规则矩阵

每条架构规则分配稳定ID，并记录：级别、规则、自动证据、人工证据、正向fixture、负向fixture、Owner、复查日和精确历史债务。

历史债务采用“只减不增”基线：新增、移动或陈旧豁免都失败。豁免消失后必须删除，不能留给未来违规复用。

### 14.3 代码审查

审查顺序：

1. 范围和授权。
2. 契约和依赖链。
3. 正常与失败行为。
4. 生命周期、并发、确定性、性能和安全。
5. 测试真实性。
6. 最终差异和生成物。

P0/P1意见必须关闭。Reviewer记录被审SHA；审查后文件变化则审查失效。

### 14.4 测试纪律

- 禁止skip/only、删除失败测试、放宽关键断言和无原因类型抑制。
- 先失败后修复，共同根因不得拆成多个局部mock掩盖。
- 真实编译、lint、测试、静态门禁分别记录；占位命令不是PASS。
- C2/C3必须给局部和全量结果及新增失败数。

### 14.5 实现方案选择

编码前至少比较：最小修复、局部重构、架构调整和不修改/延后四类方案。比较维度包括正确性、影响面、迁移成本、可逆性、测试难度、性能、安全和未来维护。

优先选择满足当前验收的最小完整方案，不选择最少代码但留下双路径的“半修复”，也不因未来可能性引入未被需求驱动的抽象。

### 14.6 Impact Map

影响图至少列出：

| 类别 | 检查内容 |
|---|---|
| Contract | 接口、类型、事件、API、schema |
| Implementation | 唯一实现、适配器、工厂 |
| Composition | 依赖注册、启动、生命周期 |
| Consumers | 直接/间接调用者、插件、脚本 |
| Persistence | 数据库、存档、缓存、序列化 |
| Configuration | 配置键、默认值、环境变量 |
| UI/Content | 文案、资源、交互、可访问性 |
| Tests | fixture、mock、契约、集成、E2E |
| Operations | 日志、指标、告警、部署、回滚 |
| Governance | 文档、进度、记忆、Skill、门禁 |

每项写`affected / not affected / unassessed`及依据。空白不等于不受影响。

### 14.7 实现过程

1. 建立最小失败复现或契约测试。
2. 修改定义方和schema。
3. 修改实现与composition root。
4. 修改全部消费者和测试fixture。
5. 删除旧路径前先证明无消费者；不能删除则显式兼容并设退出条件。
6. 运行格式、编译、架构和局部测试。
7. 检查错误处理、日志、资源释放和并发。
8. 运行全量回归并冻结SHA。

禁止边实现边不断扩大任务卡；发现设计不足时回到变更控制。

### 14.8 代码规模与复杂度

- 一个任务只解决一个共同根因。
- 公共接口数量、条件分支、依赖方向和新增文件数必须在任务卡说明。
- 超长函数、循环依赖、高扇出模块、隐式全局状态和重复实现需要独立技术债记录。
- 机械重命名、格式化和语义修改尽量拆分，便于审查。
- 生成代码不得手工维护，除非明确建立patch层。

复杂度阈值由语言/项目定义；没有工具时至少人工记录最大文件、函数、依赖和分支热点，状态写E2而非伪称静态分析PASS。

### 14.9 依赖治理

新增或升级依赖属于C2/C3，必须记录：用途、替代方案、来源、版本、许可、维护状态、已知漏洞、包大小、运行权限、传递依赖、锁文件变化和移除方案。

禁止未授权安装、浮动版本、来源不明二进制和仅为一次性脚本引入大型运行时。依赖安装后运行供应链、安全、构建和许可证检查。

### 14.10 错误处理与可观测性

- 错误必须有明确类型、上下文和恢复/失败语义。
- 禁止空catch、静默吞错和把异常转成永远成功。
- 日志包含可关联ID，不包含密钥和受限数据。
- 关键流程定义指标、告警、健康检查和降级行为。
- 重试必须有限、退避、幂等，并区分永久错误。
- 资源、连接、监听器和任务必须有生命周期释放证据。

### 14.11 测试分层

| 层级 | 目标 | 典型证据 |
|---|---|---|
| Static | 格式、lint、类型、架构 | 真实工具退出码 |
| Unit | 单函数/类和边界 | 正常、边界、错误用例 |
| Contract | 定义/实现/消费者一致 | 缺注册/不兼容负向用例 |
| Integration | 模块、数据库、外部适配 | 启动与失败链路 |
| E2E | 用户流程 | 可重复场景和环境 |
| Performance | 延迟、吞吐、内存、包体 | 基准与阈值 |
| Security | 权限、输入、秘密、依赖 | 扫描与攻击面测试 |
| Acceptance | 业务结果 | E4人工/真实环境回执 |

不同层级不能互相冒充。单元测试通过不能证明部署、序列化或真实设备正确。

### 14.12 真实编译与静态门禁

门禁必须验证命令确实调用编译器/linter，而不是echo、IDE提示或括号扫描。报告记录工具版本、配置路径、扫描范围和错误数。

引入真实工具前先只读测量全量错误，建立独立迁移计划；禁止一次性关闭严格选项、全局加ignore或批量类型断言压错。

### 14.13 缺陷处理与RCA

- 先按堆栈、错误签名和依赖链聚类失败。
- 多个失败共享根因时只建立一个根因任务。
- 修复前写“为什么现有测试/门禁没拦住”。
- 修复必须增加最小防复发证据。
- 事故级缺陷关闭后更新规则、测试或监控之一；只改代码不算完整RCA。

### 14.14 工程交付证据

交付包包含：任务卡锁、前后SHA、影响图、失败复现、实现说明、审查findings、局部/全量结果、覆盖不到的环境、性能/安全结论、回滚和未解决债务。

## 15. 配置、数据、内容和资产

本节按项目需要启用。

### 15.1 配置与数据

- 配置schema、默认值、迁移和兼容策略必须版本化。
- 禁止业务代码散落硬编码配置。
- 数据迁移必须有dry-run、备份、幂等性和回滚验证。
- 注册表与磁盘事实双向检查：磁盘未登记、登记缺文件都应报告。
- 不得在未裁决“补资源还是删登记”前自动删除记录。

### 15.2 内容与资产

每项资产至少记录：稳定ID、来源、许可、生成参数/源文件、目标规格、质量状态、注册状态、使用位置和验收证据。

文件存在不代表质量合格；占位资产、旧资产、待重做资产必须显式标记，不能算完成。

### 15.3 安全与秘密

- 密钥只能来自安全凭据系统或环境变量。
- 扫描器输出必须脱敏，不得把命中字面值写入报告。
- 当前工作区0命中不代表历史已清理。
- 密钥轮换、工作区清理和Git历史重写是三个独立动作。
- 历史重写属于C3/P0，必须协调所有开发者并单独授权。

### 15.4 配置生命周期

配置状态建议：draft→validated→active→deprecated→retired。每个配置集合记录schemaVersion、owner、environment、default policy、lastValidated和migration路径。

配置优先级必须固定，例如：安全默认值→版本化基础配置→环境配置→部署注入；禁止不受控的本地文件悄悄覆盖生产。秘密不得进入普通配置层。

### 15.5 Schema与兼容

- schema变化必须分类为兼容、可迁移不兼容或破坏性。
- 新字段定义必填/可选、默认值、空值、范围和未知字段行为。
- 删除字段前先统计消费者和数据分布。
- 读取旧版本必须明确支持窗口；写出版本必须唯一。
- schema验证器使用正反fixture，不只测试一个正常样例。

### 15.6 数据迁移

迁移计划必须包含：输入版本、输出版本、样本统计、dry-run、备份、幂等、批次、校验和、失败恢复、性能窗口、隐私影响和回滚限制。

执行顺序：冻结写入→备份验证→小样本dry-run→影子/预演→批准→分批执行→逐批校验→切换读取→观察→关闭旧路径。

不可逆迁移必须明确“回滚只能从备份恢复”，不能写模糊的“可回滚”。

### 15.7 数据分类与保留

数据分为public/internal/confidential/restricted，分别定义访问、加密、传输、日志、备份、地域和删除策略。收集必须目的限定和最小化。

每类数据有保留期和删除Owner。删除请求、法定保留和备份淘汰必须可审计；不得把“从主库删除”当成所有副本已消失。

### 15.8 资产管线

```text
source → intake review → processing → quality validation
→ registry → integration → runtime verification → release
```

每个阶段保存输入/输出hash、工具版本、Owner和状态。原始源、处理中间物、运行时产物和预览图分目录管理。

### 15.9 资产质量门

按项目定义：尺寸/格式、视觉/内容质量、性能预算、命名、透明/边界、压缩、可访问性、本地化、设备/平台兼容、来源许可和人工验收。

资产状态建议：placeholder、candidate、needs_redo、validated、integrated、deprecated。`needs_redo`不得因已注册或已进入构建就升级为完成。

### 15.10 注册表一致性

至少双向检查：磁盘存在但未登记、登记存在但文件缺失、重复ID、路径大小写、hash漂移、类型错误、消费者不存在、已废弃仍引用、生成源缺失。

修复前先裁决事实：补文件、改路径、改登记、替换资产或废弃消费者。禁止扫描器自行选择删除登记。

### 15.11 来源、许可和供应链

第三方数据/资产记录作者、来源URL/合同、许可、允许用途、署名、修改权、再分发和到期。AI生成内容记录模型/服务、提示词或工作流、输入权利和人工审核。

来源不明或许可不清的内容不得进入发布包，保持blocked。

### 15.12 秘密事件处置

1. 停止传播和日志输出。
2. 脱敏记录规则ID、路径、位置和指纹。
3. 轮换/吊销凭据；这是第一优先，不等待代码修复。
4. 工作区环境变量化和复扫。
5. 只读审计tracked/untracked/ignored/history范围。
6. 历史重写另行C3授权和协作。
7. 更新扫描器、培训和RCA。

不创建含秘密明文的报告、备份或补丁。

## 16. 文件与Git安全

### 16.1 文件操作

- 新文件放入职责明确的目录。
- 禁止凭文件名判断无用。
- 删除、移动、重命名默认dry-run。
- 破坏性操作前执行四步门：hash一致、全文引用为0、构建/脚本未引用、建立可验证安全点。
- 删除后重扫引用、构建和测试；变化异常立即停止。

### 16.2 Git规则

- 未经授权不commit、不push、不force-push、不重写历史。
- 不使用破坏性回滚清除他人未提交改动。
- 混杂工作区按任务拆分审计，不能整体提交。
- 提交应对应一个共同根因或一个可独立回滚的交付。
- 发布tag、版本号和变更日志由Release Owner批准。

### 16.3 编码、换行和路径

- 项目声明文本编码和换行策略，推荐UTF-8。
- 写入前严格解码；控制台乱码不等于文件损坏。
- 禁止U+FFFD、混合编码和不可解释BOM变化。
- 文档、脚本和配置使用仓库相对路径；运行时绝对路径只能来自环境注入。
- 跨平台检查大小写、路径分隔符、保留名称、最大长度和符号链接。
- `.gitattributes`固定需要的换行和二进制规则。

### 16.4 大文件与二进制

- 定义单文件和仓库增长阈值。
- 大型二进制使用适合的资产存储/LFS/对象存储，不反复提交压缩包。
- 二进制必须有来源、许可、hash、消费者和替换策略。
- 禁止把数据库导出、用户数据、生产日志和秘密打包入仓库。
- 发现异常增长先审计，不自动压缩或删除。

### 16.5 清理流程

```text
inventory → candidate classification → dry-run plan → reference scan
→ hash verification → safety point → explicit authorization
→ apply minimal deletion/move → rescan → test → receipt
```

候选分类：SAFE_CANDIDATE、REVIEW_REQUIRED、BLOCKED、MUST_KEEP。自动工具只能执行明确批准且所有安全门通过的SAFE_CANDIDATE。

### 16.6 提交设计

- 一个提交对应一个可解释共同根因或独立交付。
- 提交前核对实际文件集合，不使用模糊全选把混杂变化带入。
- 生成物和源修改在同一提交还是拆分，由重建和审查需要决定并记录。
- 提交信息包含task ID、目的、验证和风险，不声称未完成的结果。
- 不提交本地缓存、临时证据和敏感日志。

### 16.7 分支、合并与回退

- 分支策略必须说明生命周期、保护规则、合并方式和删除时机。
- 合并前目标分支变化需重算基线并重新验证。
- 冲突解决属于语义变更，必须审查并测试。
- 回退优先新增反向提交或发布回滚，不破坏共享历史。
- force-push、rebase共享分支和历史清理必须C3授权。

## 17. 验证门禁与完成声明

### 17.1 推荐门禁层级

1. 编码与文件完整性。
2. schema/配置验证。
3. lint/格式。
4. 真实编译/typecheck。
5. 架构规则。
6. 单元与契约测试。
7. 集成与端到端测试。
8. 资源/依赖/安全检查。
9. 文档、进度和治理一致性。
10. 真实环境或人工验收。

项目应提供一个全量入口，但报告仍需列出每个子门禁，不能只给总退出码。

### 17.2 基线失败规则

- 任务开始前测量基线。
- 新增失败必须修复。
- 失败数量不变只能说明“未引入新失败”。
- 既有失败必须登记Owner、根因、处理任务和关闭证据。
- 不能通过修改测试、忽略文件或关闭门禁降低失败数。

### 17.3 状态词

- `PASS`：真实执行且满足预期。
- `FAIL`：真实执行但不满足。
- `NOT_RUN`：工具可用但本轮未运行。
- `NOT_IMPLEMENTED`：工具/配置不存在或命令是占位。
- `BLOCKED`：存在明确阻断和解除条件。

### 17.4 门禁契约

每个门禁登记：稳定ID、名称、Owner、命令、工作目录、输入范围、前置工具、超时、PASS条件、输出位置、风险级别、可否豁免和维护触发器。

门禁必须默认只读或只写声明的派生输出。验证脚本不得静默修改被验证对象。

### 17.5 测试结果证据

记录：开始时间、环境、工具版本、命令、输入manifest SHA、退出码、通过/总数、失败标识、日志hash和结束后SHA复核。

只粘贴最后一行“passed”不足以作为E3证据；必须能追溯运行范围和失败详情。

### 17.6 历史失败基线

历史失败条目使用稳定键和精确对象：规则ID、路径/测试ID、位置或签名、Owner、原因、创建日期、复查日和关闭任务。

基线只允许减少：

- 新失败：阻断。
- 已知失败位置/签名变化：阻断并重审。
- 已知失败消失：删除基线项。
- 扩大忽略范围：独立治理变更。

### 17.7 门禁豁免

豁免必须有批准者、精确范围、到期日、补偿控制、风险和关闭任务。P0安全门、秘密明文、数据破坏和授权越界原则上不可豁免。

### 17.8 验证manifest

最终验证使用不可变manifest列出受测文件SHA、依赖锁、配置版本、工具版本和环境。manifest正文不登记自身执行后SHA；自身最终SHA放外部回执，避免自引用。

### 17.9 人工与真实环境验收

E4回执记录设备/环境、版本、场景、步骤、期望、实际、截图/日志、执行者和时间。人工“看过没问题”而无场景和版本不能作为可复核E4。

### 17.10 门禁维护

门禁出现误报、漏报、性能问题或环境不兼容时，先建立工具缺陷任务和复现fixture。不能在当前业务任务中临时关闭后继续宣布通过。

## 18. 发布、事故与回滚

### 18.1 发布条件

- 权威版本、构建输入和依赖锁已冻结。
- P0/P1为0或有Release Owner明确风险接受。
- 测试、迁移、合规、性能、安全和回滚证据齐全。
- 发布包与受测SHA一致。
- 发布后监控、告警、回滚阈值和负责人明确。

### 18.2 事故响应

```text
停止扩散 → 保存证据 → 评估影响 → 回滚/隔离
→ 修复共同根因 → 全量验证 → RCA → 更新门禁
```

RCA至少记录：现象、影响、时间线、直接原因、促成条件、为何门禁未发现、排除假设、修复边界、防复发措施、Owner和关闭证据。

### 18.3 环境与晋级

建议分：local/dev→test→staging→production。每次晋级绑定相同构建产物hash，不在环境间重新手工构建。

环境差异必须版本化或由受控变量注入。生产访问最小权限，人工临时修改必须记录并回写事实源，防止配置漂移。

### 18.4 版本与发布包

- 采用明确版本策略并记录兼容范围。
- 发布包包含构建manifest、依赖锁、变更日志、迁移、验证和回滚入口。
- 版本号、tag和包hash一致。
- 可重复构建项目应验证同输入得到等价输出；不可重复部分必须说明原因。
- 候选发布包冻结后禁止手工替换单个文件。

### 18.5 发布步骤

1. 确认发布范围和Release Owner。
2. 冻结代码、配置、依赖和资产manifest。
3. 完成全量门禁、E4和风险复核。
4. 验证备份、迁移和回滚。
5. 创建唯一发布候选并计算hash。
6. 受控部署，记录请求ID/环境/时间。
7. 执行烟雾测试、指标观察和业务验收。
8. 达到阈值则扩大流量；异常则停止或回滚。
9. 更新发布记录和进度。

### 18.6 回滚策略

回滚分为应用版本回滚、配置回滚、流量切换、功能开关、数据恢复和前向修复。每种方式明确适用条件、数据兼容和最大恢复时间。

发布前必须演练关键回滚路径。涉及不可逆数据变化时，应用回滚可能不可用，必须采用双写、兼容读取或备份恢复方案。

### 18.7 事故等级

| 等级 | 示例 | 响应 |
|---|---|---|
| SEV0 | 安全泄漏、广泛数据损坏、重大合规事件 | 立即止血、最高负责人、持续响应 |
| SEV1 | 核心功能不可用、重大用户影响 | 立即响应和回滚决策 |
| SEV2 | 部分功能降级、有替代路径 | 当日处理和监控 |
| SEV3 | 轻微问题、无显著用户影响 | 正常缺陷流程 |

### 18.8 事故指挥与沟通

指定Incident Commander、Operations、Communications和Scribe。执行者不同时对外随意发布不同结论。沟通只给已验证事实、影响、缓解、下一更新时间，不泄漏秘密或未经证实根因。

### 18.9 事后复盘

复盘不追责个人，关注系统条件。行动项必须有Owner、优先级、截止日和验证；“加强注意”不是合格防复发措施。RCA关闭前检查行动项是否进入路线图和门禁。

## 19. 管理节奏与指标

### 19.1 节奏

- 每个任务：启动回执、checkpoint、handoff、验收。
- 每日或每个工作窗口：活动任务、阻断、租约和热文件检查。
- 每周或里程碑：权威索引、进度、风险、记忆、Skill、依赖和技术债复查。
- 每次发布前：冻结写入、全量验证、安全/合规检查和回滚演练。
- 每次事故后：RCA和规则/测试演进。

### 19.2 推荐指标

| 领域 | 指标 |
|---|---|
| 文档 | current冲突、坏链接、缺元数据、过期文档 |
| 进度 | unassessed、blocked、无证据in_progress、状态漂移 |
| 多AI | 文件冲突、SHA漂移、过期租约、无handoff任务 |
| 记忆 | 活动根数量、待合并补丁、过期条目、冲突ID、秘密命中 |
| Skill | 未登记能力、安装漂移、隔离项、缺权限声明、触发冲突 |
| 工程 | 测试通过率、新增失败、伪typecheck、架构债务、开放P0/P1 |
| 发布 | 回滚次数、变更失败率、恢复时间、未经验证发布数 |

指标用于发现风险，不用于逼迫执行者伪造绿色状态。

### 19.3 每日/工作窗口检查

- active任务、Owner、租约和最后checkpoint。
- P0/P1、blocked和即将过期例外。
- 热文件和集成队列。
- 新增失败、外部依赖故障和安全告警。
- 即将发布或需要人工决策的事项。

输出简短状态，不复制整份任务记录。

### 19.4 每周/里程碑评审

1. 权威索引和current新鲜度。
2. 路线图依赖、unassessed和长期blocked。
3. 目录漂移、根层散落和生成物增长。
4. 记忆补丁、过期事实和容量。
5. Skill漂移、隔离和权限。
6. 测试趋势、架构债务、依赖和安全漏洞。
7. 发布准备、事故行动项和回滚能力。

### 19.5 指标定义要求

每个指标必须说明：目的、公式、数据源、Owner、更新频率、阈值、例外和可能被误用的方式。没有稳定数据源的指标标E0/E1，不制作虚假精确仪表盘。

### 19.6 健康阈值示例

| 指标 | 绿色 | 黄色 | 红色 |
|---|---:|---:|---:|
| current冲突 | 0 | — | >0 |
| 新增门禁失败 | 0 | — | >0 |
| 无Owner的P0/P1 | 0 | — | >0 |
| 活动根数量 | 1 | — | ≠1 |
| 同路径并发Writer | 0 | — | >0 |
| 过期C3例外 | 0 | 1 | >1 |
| 无handoff长任务 | 0 | 1 | >1 |

具体数值按项目校准；安全和权威冲突通常不设“可接受黄色”。

### 19.7 审计与抽样

每个里程碑随机抽查若干任务，验证授权、Change List、SHA、checkpoint、审查、测试和完成声明是否一致。发现系统性缺口时更新模板/门禁，不只修单个报告。

### 19.8 治理成本控制

- C0/C1使用轻量模板，C2/C3使用完整流程。
- 机器可推导的信息自动生成，不要求人工重复填写。
- 同一事实只维护一份机器源，其他视图生成。
- 定期删除过期租约和派生缓存需走既定清理流程，但保留审计历史。
- 流程步骤若长期不产生决策或证据，应通过ADR评估简化，而不是私下绕过。

## 20. 落地路径

### Phase 0：只读基线

- 盘点文件、文档、测试、构建、Git、记忆、Skill、依赖和秘密风险。
- 运行现有可用门禁，不修复、不美化结果。
- 输出基线报告和未决裁决。

**输入**：现有仓库/工作区、工具清单、负责人访谈。  
**交付**：文件/文档/门禁/风险基线、敏感扫描、未决决策表。  
**完成门**：所有结论带证据等级；工具不可用项标NOT_RUN/NOT_IMPLEMENTED；没有业务写入。

### Phase 1：权威与入口

- 创建`PROJECT_RULES.md`和权威索引。
- 每领域指定唯一current或明确`unassessed`。
- 标记historical/superseded/deprecated关系。

**输入**：基线报告和Decision Owner裁决。  
**交付**：项目入口、权威索引、术语表、ADR索引、文档元数据规则。  
**完成门**：current重复=0；未裁定领域显式unassessed；入口能路由所有核心规则。

### Phase 2：任务与进度

- 冻结七值状态、任务卡、Change List、TASK_STATE、checkpoint和handoff格式。
- 建立总体进度索引。

**输入**：权威领域和现有路线图。  
**交付**：状态词表、进度schema、任务卡、风险表、总体索引和检查器。  
**完成门**：无非法状态/空壳；个体状态与索引计数一致；blocked有解除条件。

### Phase 3：多AI与上下文恢复

- 建立协调目录、文件声明、SHA基线、唯一Integrator和恢复门禁。
- 将大任务拆为可独立验收的小批次。

**输入**：任务模型、文件分级和团队模式。  
**交付**：协调目录、租约schema、baseline/checkpoint/handoff模板、冲突检查器。  
**完成门**：重叠路径能阻断；新会话恢复演练通过；只有Integrator可集成热文件。

### Phase 4：记忆和Skill

- 确定唯一活动记忆根。
- 建立Skill注册表、安装锁、触发矩阵和隔离流程。

**输入**：全部记忆根、Skill/插件/自动化库存。  
**交付**：根裁决、记忆schema/runbook、Skill注册表/锁/安全审查。  
**完成门**：活动记忆根=1；双active=0；hash漂移被隔离；秘密与冲突可阻断。

### Phase 5：工程、安全和自动门禁

- 建立工程基线、架构规则矩阵、测试策略、审查和RCA。
- 把重要治理规则转为只读检查器和负向测试。

**输入**：现有代码/内容/数据质量基线和事故历史。  
**交付**：工程任务模板、架构矩阵、测试策略、RCA、文件放置/安全/验证门禁。  
**完成门**：规则有稳定ID和负向fixture；历史债务只减不增；伪门禁不可报告PASS。

### Phase 6：终验与持续维护

- 冻结写入后运行全量验证。
- 对比整改前后指标。
- 如实记录未解决项；验证失败时不得宣称全部完成。

**输入**：Phase 0基线、所有阶段报告、冻结manifest。  
**交付**：最终报告、前后指标、开放阻断、维护节奏和责任表。  
**完成门**：新增失败=0；风险完整披露；受测SHA未变化；验收状态由授权者决定。

### 20.1 阶段执行纪律

- 每个Phase独立Change List、报告和验收，不把全项目一次性提交。
- Phase之间的依赖不满足时保持blocked，不跳阶段伪造后续产物。
- 可并行的只读盘点可并行，current、索引、共享门禁和最终报告串行。
- 每阶段先建立检查器的负向fixture，再宣布规则已自动化。
- 历史失败可以留存，但必须在Phase 6明确Owner和后续任务。

### 20.2 三种采用规模

| 规模 | 适用 | 最小内容 |
|---|---|---|
| Lite | 单人原型、短周期、低敏感 | 入口、权威索引、任务卡、TASK_STATE、测试、Git安全 |
| Standard | 持续开发、人机混合 | 全部核心章节、协调、进度、记忆、Skill、工程门禁 |
| Regulated/Critical | 生产数据、合规、不可逆操作 | Standard + 职责分离、审计、E4、发布/回滚演练、正式风险接受 |

规模只影响文档和审查强度，不降低秘密、数据、删除和发布底线。

### 20.3 迁移旧项目的原则

- 先建立基线和只减不增规则，避免一次批量移动造成大面积冲突。
- 优先治理入口、热文件、秘密、删除安全和测试真实性。
- 旧文档先分类，不急于重写全部内容。
- 旧目录先建立placement规则和候选清单，再按小批迁移。
- 每批保持可回滚，不用一次“整理仓库”混合几百项变化。

## 22. 方案验收标准

一个项目只有在以下条件满足后，才算真正采用本方案：

- 有唯一项目入口规则和领域权威索引。
- current冲突能被发现并阻断。
- 每个写任务有精确范围和执行前SHA。
- 长任务能在新对话中根据磁盘状态无歧义恢复。
- 多AI不会同时写同一文件，共享热文件只有唯一Integrator。
- 进度状态有证据且计数一致。
- 记忆只有一个活动根，Skill有注册表和安装锁。
- C2/C3代码任务有影响图、失败证据和独立审查。
- 删除、迁移、密钥、发布等高风险操作有独立安全门。
- 全量验证可以区分PASS、FAIL、NOT_RUN和NOT_IMPLEMENTED。
- 执行者不能自行将自己的交付标为accepted。
- 验证失败时，报告会明确写“不通过”，而不是为了结束任务宣称完成。

本方案的成功标准不是“流程文件很多”，而是任何新成员或新AI都能在不依赖旧对话的情况下，准确知道当前事实、允许范围、下一动作、验证方法和停止条件。

### 22.1 成熟度模型

| 等级 | 特征 |
|---|---|
| L0 Ad hoc | 依赖聊天和个人记忆，无权威入口 |
| L1 Visible | 有入口、任务和基础状态，但多为人工维护 |
| L2 Controlled | 有SHA冻结、角色、文件声明、审查和统一门禁 |
| L3 Automated | 核心规则有机器检查和负向fixture，状态由机器源生成 |
| L4 Adaptive | 指标、事故和审计持续驱动规则演进，治理成本受控 |

项目不必所有领域同时达到同一级别。P0安全和发布应优先达到L2/L3。

### 22.2 验收审计表

| 领域 | 核心问题 | 通过条件 |
|---|---|---|
| 入口 | 新AI从哪里开始 | 唯一入口可路由全部规则 |
| 权威 | 当前决定在哪里 | 每领域≤1 current |
| 文件 | 新文件放哪里 | placement规则和根allowlist有效 |
| 任务 | 允许改什么 | Change List+SHA完整 |
| 恢复 | 上下文丢失怎么办 | 新会话恢复演练通过 |
| 多AI | 谁写哪个文件 | 重叠Writer=0 |
| 进度 | 状态是否有证据 | 非法/无证据状态=0 |
| 记忆 | 长期事实谁维护 | 活动根=1，合并独占 |
| Skill | 工具是否可信 | active+available+hash匹配 |
| 工程 | 是否修共同根因 | 契约/影响/测试/审查完整 |
| 数据资产 | 是否可追溯 | schema/来源/注册/质量证据齐全 |
| 安全 | 是否泄漏或越权 | 秘密、权限、删除门禁有效 |
| 验证 | PASS是否真实 | 命令、输入SHA、退出码可追溯 |
| 发布 | 受测即发布吗 | 包hash一致，回滚可用 |

### 22.3 反模式检查

出现以下任一情况说明方案尚未真正落地：

- 规则只存在于长文档，没有入口和检查器。
- 每次换AI都要重新解释项目历史。
- 任务状态只在聊天中，无法从磁盘恢复。
- 文件不断出现`final_v3_new`和多个current。
- 多AI通过“都改完再合并”协作。
- 测试失败被称为预存后永久无人负责。
- Skill/脚本从用户目录直接运行而无版本锁。
- 记忆文件同时有多个活动副本。
- Reviewer审查后Executor继续改文件但不重审。
- 发布包与测试对象不是同一hash。
- 清理工具根据文件名自动删除。
- 报告由执行AI自行改为accepted。

### 22.4 方案自身维护

本方案也受其规则约束。修改时必须：

1. 建立独立Change List和版本锁。
2. 说明变更针对的失败或管理缺口。
3. 评估对模板、检查器和现有任务的影响。
4. 更新版本和变更记录。
5. 运行结构、链接、状态和示例检查。
6. 不在业务任务中顺手降低治理要求。

建议每季度、重大事故后、团队/AI工具变化后和发布模式变化后复查。
